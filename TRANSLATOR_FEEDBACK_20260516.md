# 번역가의 역할 및 피드백

## 1️⃣ 번역가가 받아야 할 정보 (정보 수집 시스템)

### 현재 제안 (문서 Section 2)
> "번역가 채널: Medium 기술 기사 추천 + 한영 기술 용어 표준화 + 번역 품질 검증 체크리스트"

### 번역가의 피드백

**✅ 수동할 수 있는 부분:**
1. **Medium 기술 기사 추천** (한영 번역 학습용)
   - 예: React 성능 최적화, TypeScript 타입 안전성, Next.js 마이그레이션 가이드
   - 포맷: 기사 제목 + URL + 한 줄 요약 (AI 생성 가능)
   - 주 1회, 3~5개 기사 (처리 가능)

2. **한영 기술 용어 표준화 추천안**
   - 예: API Response Time → "API 응답 시간" (O) vs "API 응답속도" (X)
   - 포맷: 영어 용어 | 추천 한국어 | 대안 | 주의사항
   - 주 1회, 5~10개 신규 용어 (가능)
   - 참고: GitHub, 한국 개발자 커뮤니티 등에서 수집

3. **번역 품질 검증 체크리스트**
   - 예: "이 번역이 정확한가?", "더 나은 표현이 있나?"
   - 포맷: 체크리스트 (기술 정확도, 문맥 적절성, 일관성)
   - 주 1회 (팀원 번역 문서 2~3개 검증)

**⚠️ 조건:**
- 포트폴리오 개발과 **병렬 가능**하려면, 정보가 **사전 정제되어야 함** (숫자, 링크 등)
- 따라서 자동 정보 수집 에이전트가 **이미 필터링한 상태**에서 번역가가 받아야 함
- "번역가가 직접 Medium 방문하고 찾기" ❌ → "정제된 기사 3개가 미리 도착" ✅

## 2️⃣ 기술 용어 표준화 (Terminology Management)

### 현재 문제점

1. **용어 일관성 부족**
   - 팀원마다 다르게 쓰는 용어: "API 응답 시간" vs "API 응답속도" vs "API 반응 속도"
   - 포탈 문서 곳곳에 산재
   - 번역 정확도 기준에 영향

2. **월간 기준 변경 시 용어 변동 리스크**
   - 평가 기준 동적화 시스템이 매월 새 용어 도입 가능
   - 예: 5월 "Core Web Vitals" → 6월 "Interaction to Next Paint" 추가
   - 팀원들이 혼동할 가능성

3. **한영 대역 불균형**
   - 영어 기술 기사 → 한국어 설명서로 자동 변환될 때 오류 발생
   - 예: "Latency" → "지연 시간" vs "지연" vs "레이턴시"

### 번역가 제안: 용어 표준화 시스템

#### **안건 1: 팀 용어 가이드 (Glossary) 구축**

**포맷:** `TERMINOLOGY_GLOSSARY.md` (월간 갱신)
```markdown
# 2026-05 팀 기술 용어 표준안

## API & 성능
| 영어 | 추천 한국어 | 대안 | 배경 |
|------|-----------|------|------|
| API Response Time | API 응답 시간 | API 반응속도 | 평가 기준 <150ms 관련 |
| Latency | 지연 시간 | 레이턴시 | Supabase 최적화 관련 |
| Throughput | 처리량 | 처리 속도 | 백업 메트릭 관련 |
| Core Web Vitals | 핵심 웹 생명력 | 코어 웹 바이탈 | Google 권장 |

## 번역 & QA
| 영어 | 추천 한국어 | 대안 | 배경 |
|------|-----------|------|------|
| Accuracy | 정확도 | 정확성 | 평가 기준 96% 관련 |
| Consistency | 일관성 | 일관된 성 | 용어 표준화 관련 |
| Validation | 검증 | 확인 | 품질 관리 관련 |

## 개발 & 운영
| 영어 | 추천 한국어 | 대안 | 배경 |
|------|-----------|------|------|
| Deployment | 배포 | 디플로이 | Vercel 관련 |
| Build Time | 빌드 시간 | 빌드 소요시간 | 성능 기준 관련 |
| Rollback | 롤백 | 복구 | 배포 실패 관련 |
```

**운영 규칙:**
- 매월 15일 평가 기준 동적화 시 **동시에 용어 기준도 업데이트**
- 번역가가 월간 팀 상의에서 "신규 용어 추가/변경" 1가지 제안
- Git 커밋: `chore(terminology): 2026-05 용어 기준 업데이트`

**효과:**
- 팀 전체가 같은 용어 사용 (일관성 99%+ 목표)
- 번역 정확도 향상
- 새 팀원 온보딩 시 가이드 제공

2. **향후 리스크**
   - 평가 기준 동적화 때마다 새 영어 용어 추가 → 한국어 표현 불명확

   - 용어 일관성이 필수 (현재 95%일 때 용어 오류만 5%)
   - Glossary가 있으면 자동 검증 가능

### 권장 안내

**1단계 (즉시):** `TERMINOLOGY_GLOSSARY.md` 생성
- 핵심 20개 용어 정의
- 예: API, Deployment, Accuracy, Latency, Consistency, Validation, Rollback, Build Time

**2단계 (2주 후):** 모든 기존 문서에 용어 적용
- 자동 검색 & 대체 (필요시 수동)
- 예: "응답 시간" 전부 → "API 응답 시간"으로 통일

**3단계 (월간):** 평가 기준 동적화와 동시에 용어 업데이트
- 새 기준 추가 → 새 용어도 함께 정의
