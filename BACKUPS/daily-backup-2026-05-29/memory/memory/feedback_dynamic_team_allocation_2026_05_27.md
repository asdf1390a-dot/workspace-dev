## 규칙

**절대:** 팀원이 외부 블로킹(사용자 액션, 의존성 대기)을 기다리게 하지 말 것.
- 그 팀원을 백그라운드로 빼두고 (신호 감시 자동)
- 다른 팀원을 즉시 새 작업에 배치

**블로킹 풀렸을 때:**
- 자동으로 그 팀원이 원래 작업 복귀
- 진행 중인 새 작업은 영향 없음 (병렬 유지)

**결과:** 용량 활용률 증가, 대기 시간 0

## 예시

**현재 (경직됨):**
- BM-P1 평가자: 평가 신호 대기 중 → 그냥 기다림
- 결과: 8/12명만 활용

**개선 (유연함):**
- BM-P1 평가자: 평가 신호 감시 (백그라운드, 자동)
- 평가자 역할: 즉시 다른 프로젝트(예: Design QA) 배치
- 블로킹 풀림 → 평가자 원래 작업 복귀 (Design QA 일시 중지)
- 결과: 12/12명 활용, 대기 0

## 적용 대상

- 모든 외부 블로킹: 사용자 액션, API 응답 대기, 코드 리뷰 신호 등
- 모든 팀원: 비서 포함

## 좋은 점

- "똑똑해짐" — 비서가 자동으로 판단해서 팀원 재배치
- 블로킹 당사자는 신경 쓸 필요 없음 (자동 복귀)
- 용량 낭비 제거

**누가 말함:** CEO 2026-05-27 19:31 KST ("왜 8명만일해...규칙이야 좀똑똑해져라 비서야")
**Why:** 12명 팀을 제대로 활용하려면 병렬 처리가 필수. 블로킹은 기회, 낭비 아님.
