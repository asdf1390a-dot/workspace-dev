# 설계 문서 워크플로우 — 평가자 검토 게이트

**최종 수정:** 2026-05-25 (Rule Conflict Resolution Phase 1)

**핵심 변경:** "설계 = 진행 신호" → "설계 = 평가자 검토 신호"
- 모든 설계 문서는 평가자 검토를 거친 후 구현으로 이동
- 단, 설계 타입에 따라 검토 기준과 타이밍 다름

---

## 1️⃣ 설계 타입 분류 (Design Type Classification)

### 타입 A: 사용자 대면 UI (User-Facing UI)
**대상:** Asset Master, Backup App, Travel Management, 경영실적 대시보드 등

**설계 → 검토 → 구현 흐름:**
```
플레너가 설계 완료 (*_DESIGN.md)
  ↓
【평가자에게 자동 위임】검토 체크리스트:
  - 기능 명세 명확? (누락 없음)
  - UI/UX 일관성? (색상, 간격, 폰트)
  - 접근성? (WCAG AA 기준)
  - 성능 목표? (초기로딩 <3s)
  ↓ ✅ 승인
【웹개발자 구현 시작】DB → API → UI 순
```

**검토 기준:**
- 기능 정의서 명확도
- 화면 설계 (mockup/wireframe)
- 상태 관리 흐름도
- 오류 처리 시나리오

**타이밍:** 설계 완료 후 최소 1-2시간 내 평가자 검토 시작

---

### 타입 B: 백엔드 API/DB (Backend API & Database)
**대상:** PM Phase 1, BM Phase 1, Asset Master API v2 등

**설계 → 병렬 검토 + 구현 흐름:**
```
플레너가 설계 완료 (*_DESIGN.md)
  ↓
【평가자에게 위임】가벼운 검토 (1-2분):
  - DB 스키마 충돌 없나? (중복 테이블명, 컬럼 이름 겹침)
  - API 명세 문법 맞나? (경로, 파라미터 형식)
  ↓ ✅ 통과
【웹개발자 즉시 구현 가능】(병렬 진행)
  DB 마이그레이션 작성 → SQL 푸시
```

**병렬 처리 규칙:**
- 설계 완료 → 평가자 경량 검토 (동기: 오류 감지만)
- 웹개발자는 동시에 구현 시작 (설계 완료 = 충분한 정보)
- 평가자의 상세 검토는 구현 진행 중 병렬 처리
- 결함 발견 시만 → 웹개발자에게 피드백 및 수정

**검토 기준:**
- 테이블명 중복 확인
- 컬럼명 일관성
- API 경로 충돌 (동일 메소드+경로 금지)
- 데이터 타입 호환성

**타이밍:** 설계 완료 직후 5-10분 내 경량 검토 → 웹개발자 구현 시작

---

### 타입 C: 자동화 스크립트 (Automation & Cron Jobs)
**대상:** Hermes 백그라운드 Job, CTB Cron 감시, 자동 리포트 생성 등

**설계 = 즉시 실행 신호:**
```
플레너가 설계 완료 (*_DESIGN.md)
  ↓
【평가자 우회】설계 완료 = 실행 신호
  ↓
【데이터분석가 또는 자동화전문가 즉시 구현】
  (설계의 명확도가 높으면 평가자 사전검토 생략 가능)
```

**규칙:**
- 자동화 스크립트는 사용자 인터페이스 없음 → 설계만 명확하면 즉시 구현
- 평가자 검토는 실행 후 결과 검증 형태로 진행
- 단, 중요 데이터 변경 (DELETE, UPDATE대량) 있으면 → 사전 검토 필수

**검토 기준:**
- 일정 트리거 명확? (cron 표현식)
- 데이터 입력 경로 명확? (어느 테이블에서 읽나)
- 출력 형식 정의? (로그, DB 업데이트, Telegram 메시지)
- 에러 핸들링 정의? (실패 시 재시도, 알림)

**타이밍:** 설계 완료 즉시 구현 시작

## 2️⃣ 설계 문서 메타데이터 필드 (Design Document Metadata)

**모든 설계 문서에 frontmatter 추가 (2026-05-25부터):**

```markdown
---
name: [프로젝트명] 설계
design_type: A | B | C  # 위 분류 중 하나
evaluation_required: true | false
evaluation_gate: "reviewer" | "lightweight" | "skip"
expected_review_time: "1-2h" | "5-10min" | "none"
owner_next_stage: web-builder | data-analyst | automation-specialist
---
```

**예시:**

```markdown
---
name: Asset Master Phase 2 설계
design_type: A
evaluation_required: true
evaluation_gate: "reviewer"
expected_review_time: "2h"
owner_next_stage: web-builder
---
```

## 3️⃣ 각 설계 타입별 평가 체크리스트

### 타입 A: UI 검토 체크리스트
- [ ] 기능 명세가 완전한가? (빠진 상호작용 없음)
- [ ] 모든 화면이 wireframe/mockup으로 표시되었는가?
- [ ] 상태 전환이 명확한가? (e.g., 로딩 → 완료 → 에러)
- [ ] 폼 검증 규칙이 정의되었는가? (필수/선택, 포맷)
- [ ] 오류 메시지가 구체적인가? ("실패했습니다" ❌ vs "최대 5개 파일만 업로드 가능" ✅)
- [ ] 접근성 기준이 충족되는가? (WCAG AA 색상 대비)
- [ ] 성능 목표가 수립되었는가? (첫 로딩 < 3초 등)

### 타입 B: API 경량 검토 체크리스트
- [ ] 테이블명 중복이 없는가? (기존 테이블과 이름 안 겹침)
- [ ] API 경로 충돌이 없는가? (POST /api/assets vs POST /api/assets)
- [ ] 데이터 타입이 일관적인가? (id는 항상 uuid 등)
- [ ] 필수/선택 파라미터가 명확한가?

### 타입 C: 자동화 검수 체크리스트
- [ ] Cron 일정이 유효한가? (cron 표현식 문법)
- [ ] 데이터 입력 경로가 존재하는가? (테이블, 파일 경로 등)
- [ ] 실패 시 재시도 전략이 있는가?
- [ ] 알림/로깅이 정의되었는가?

## 4️⃣ 실행 플로우 (Execution Flow)

### 플레너의 책임
1. 설계 문서 작성 완료
2. 메타데이터 필드 추가 (design_type, evaluation_required, 등)
3. frontmatter 포함 상태로 저장
4. **자동으로** 평가자(또는 해당 팀원)에게 위임

### 평가자의 책임
- **타입 A UI:** 상세 검토 (1-2시간)
- **타입 B API:** 경량 검토 (5-10분)
- **타입 C 자동화:** 스킵 (또는 구현 후 결과 검증)

### 웹개발자의 책임
- 타입 A: 평가자 ✅ 승인 후 구현 시작
- 타입 B: 설계 완료 직후 즉시 구현 (평가자 경량검토와 병렬)
- 타입 C: 자동화전문가/데이터분석가가 구현

---

## 5️⃣ 상태 전환 (State Transitions)

```
설계 완료 (DESIGN_COMPLETE)
  ↓ [타입 A/B]
평가자 검토 중 (EVALUATION_IN_PROGRESS) ← 약속 시간 설정
  ↓ ✅
평가자 승인 (EVALUATION_APPROVED)
  ↓ [타입 A]
구현 시작 (IMPLEMENTATION_READY) ← 웹개발자에게 신호
  
[타입 B: 병렬 처리]
  설계 완료 → 웹개발자 즉시 구현 + 평가자 경량검토 병렬
  
[타입 C: 직접 실행]
  설계 완료 → 자동화전문가 즉시 구현 → 실행 후 평가자 검증
```
