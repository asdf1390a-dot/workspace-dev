## 규칙

**SQL 쿼리 작성 시 절대 준수 규칙:**

1. **선행 단계: 스키마 확인 (MUST)**
   - 실제 테이블 존재 확인
   - information_schema 또는 pg_* 시스템 테이블의 정확한 컬럼명 사전 조회
   - 가정/기억으로 컬럼명 사용 금지

2. **쿼리 작성 단계**
   - 확인된 정보만으로 쿼리 작성
   - 예: `pg_trigger.trigname` 아니라 `pg_trigger.tgname` 사용

3. **자체 검증 3단계 (발송 전)**
   - ✅ 테이블/컬럼명 존재 확인
   - ✅ 문법 검증 (WHERE 절 로직, JOIN 조건 확인)
   - ✅ 예상 결과 문서화 (사용자가 뭘 봐야 하는지 명시)
   - 이 3단계를 모두 통과해야만 사용자에게 발송

4. **발송 이전 체크리스트**
   ```
   - [ ] 테이블/컬럼명 사전 조회 완료
   - [ ] 쿼리 문법 검증 완료
   - [ ] 사용자 예상 결과 문서화 완료
   - [ ] 링크 + 코드 + 단계 형식 검증 완료
   ```

## Why (이전 위반)

2026-05-15 Asset Master Phase 1 검증 쿼리에서:
- `pg_trigger.trigname` → 존재하지 않는 컬럼 (정답: `tgname`)
- `t.trigname does not exist` 오류 발생
- 사용자가 3회 같은 실수로 지적

**근본 원인:** 
- Postgres 문서를 미리 확인하지 않고 일반적인 네이밍 규칙으로 가정
- 작은 쿼리라고 생각해 자체 검증 생략
- 발송 전 "이게 맞나?" 한 번도 안 함

## How to apply

**즉시 (매번):**
1. SQL 쿼리 작성 전 → 시스템 테이블/뷰 정확한 구조 확인
2. 쿼리 작성 후 → 자체 검증 3단계 수행
3. 발송 전 → 위 체크리스트 4항 모두 통과 확인

**평가자 리뷰 추가 (5월 22일부터):**
- 웹/데이터 분석 작업에서 SQL 발송 전 평가자 사전 검토 단계 추가
- 체크리스트: "테이블 존재? → 컬럼명 정확? → 문법 OK? → 실행 가능한가?"
