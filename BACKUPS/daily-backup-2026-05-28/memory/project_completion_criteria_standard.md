## 핵심 규칙

**완료 기준이 없는 "완료" 주장은 받지 않음**
- 모든 작업은 사전에 완료 기준 정의 필요
- 기준 = 기술 세부사항 기반 (코드, 배포, 테스트, 검증 등)
- 모호한 표현("했습니다", "완료했습니다") 절대 금지

**표준 완료 기준 (단계별)**

### 1. 설계 (DESIGN) 완료

❌ 잘못된 완료:
- "설계 문서 작성했습니다"
- "아키텍처 정했습니다"

✅ 올바른 완료 기준:
- 📄 설계 문서 완성 (3개 산출물: _DESIGN.md, _PROPOSAL.md, _PLAN.md)
- 📋 구현 체크리스트 작성 (20+ 항목, 각 항목당 파일경로 명시)
- 🗂️ DB 스키마 정의 (CREATE TABLE 포함)
- 🔌 API 명세 (엔드포인트, 요청/응답, 에러 코드 명시)
- 🎨 UI/UX 디자인 (와이어프레임, 컴포넌트 목록)
- ✅ 팀 리뷰 완료 (평가자 승인)

**기록 방식:** Git commit message에 `Stage: DESIGN` 명시

---

### 2. DB 스키마 (DB) 완료

❌ 잘못된 완료:
- "테이블 만들었습니다"
- "마이그레이션 했습니다"

✅ 올바른 완료 기준:
- 📝 SQL 마이그레이션 파일 작성 (db/NN_*.sql 형식)
- 🔐 RLS 정책 정의 (모든 테이블)
- 📊 인덱스 최적화 (성능 고려)
- ✅ Supabase에 직접 적용 확인 (수동 또는 자동화)
- 🧪 롤백 계획 작성 (문제 발생 시 복구 방법)
- 📋 테이블 메타데이터 정의 (컬럼 설명, 제약조건)

**기록 방식:**
- Git commit message에 `Stage: DB` + 마이그레이션 파일 경로 명시
- CTB 기록: commit 해시 + 테이블 목록 + RLS 정책 상태

---

### 3. API 구현 (API) 완료

❌ 잘못된 완료:
- "API 만들었습니다"
- "엔드포인트 추가했습니다"

✅ 올바른 완료 기준:
- 📝 모든 엔드포인트 구현 (GET, POST, PUT, DELETE)
- 🔐 인증/권한 검증 추가
- 🧪 E2E 테스트 통과 (Postman 또는 자동화)
- 📊 에러 처리 (모든 에러 케이스에 대해 상태 코드 + 메시지)
- 📋 API 문서화 (요청/응답 예시 포함)
- ✅ 로컬 개발 서버 테스트 완료 (npm run dev)
- 🚀 Vercel 빌드 성공 (배포 가능 상태)

**기록 방식:**
- Git commit message에 `Stage: API` + 엔드포인트 개수 명시
- CTB 기록: commit 해시 + API 목록 + E2E 테스트 결과

---

### 4. UI 구현 (UI) 완료

❌ 잘못된 완료:
- "화면 만들었습니다"
- "컴포넌트 추가했습니다"

✅ 올바른 완료 기준:
- 📝 모든 화면 페이지 구현
- 🎨 CSS 스타일 완성 (반응형 디자인 포함, 모바일 320px ~ 데스크톱 1920px)
- 📋 폼 검증 추가 (모든 입력 필드)
- 🔌 API 통합 완료 (모든 CRUD 동작)
- 🧪 로컬 테스트 통과 (npm run dev에서 전체 플로우 동작)
- 📊 로딩 상태/에러 상태 UI 구현 (Loading, Error, Empty 모두)
- ✅ 접근성 검증 (WCAG AA 최소 기준)
- 🎬 비즈니스 로직 검증 (사용자 관점에서 기능 동작 확인)

**기록 방식:**
- Git commit message에 `Stage: UI` + 페이지/컴포넌트 개수 명시
- CTB 기록: commit 해시 + 페이지 목록 + 테스트 결과

---

### 5. 배포 (DEPLOY) 완료

❌ 잘못된 완료:
- "배포했습니다"
- "푸시했습니다"

✅ 올바른 완료 기준:
- 📝 모든 코드 커밋 완료 (로컬 브랜치 → GitHub)
- ✅ GitHub 푸시 성공 확인 (commit 해시 기록)
- 🚀 Vercel 빌드 성공 (Preview 또는 Production)
- 🌐 라이브 사이트에서 접속 가능 (https://dsc-fms-portal.vercel.app)
- 📊 메인 페이지 로딩 확인 (HTTP 200)
- 🧪 배포 후 스모크 테스트 (기본 기능 5개 이상)

**기록 방식:**
- Git commit message에 `Stage: DEPLOY`
- CTB 기록: commit 해시 + Vercel 배포 URL + 스모크 테스트 결과

---

### 6. 검증 (VERIFY) 완료

❌ 잘못된 완료:
- "테스트했습니다"
- "확인했습니다"

✅ 올바른 완료 기준:
- 📝 평가자(evaluator) 검증 통과
  - 최소 3회 반복 테스트 (동일 기능 여러 케이스)
  - 버그/결함 발견 → 즉시 개발자 피드백 → 수정 → 재검증
  - 최종 승인 전 모든 결함 해결
- 📊 자동화 테스트 통과 (있는 경우)
- 🎬 사용자 승인 (해당하는 경우)
- 📋 최종 리포트 작성 (통과/실패 사항, 적용 시기)

**기록 방식:**
- CTB 기록: evaluator 검증 결과 + 발견된 버그 + 수정 내역 + 최종 승인 날짜
