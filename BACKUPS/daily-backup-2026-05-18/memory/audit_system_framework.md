## 개요

**목표:** 팀 업무(API 구현, 설계, 배포, 품질보증)의 신뢰도를 매일 평가하고, 발견된 문제를 자동으로 개선하는 체계

**신뢰도 정의:**
- ✅ 약속한 산출물이 제때 나오는가?
- ✅ 산출물의 품질이 요구사항을 만족하는가?
- ✅ 팀 간 정보 흐름(memory, task tracking)이 정확한가?
- ✅ 의존성 관리와 우선순위 판단이 적절한가?

---

## 1. 일일 감시 지표 (Daily Audit Metrics)

### 1.1 코드 품질 지표
| 지표 | 목표 | 측정 방법 |
|-----|-----|---------|
| Build Success Rate | 100% | npm run build, 배포 상태 |
| API Test Pass Rate | ≥95% | 평가자 검증 결과 |
| Deploy Availability | 100% | Vercel HTTP 200, API 응답 |
| CTB 정확성 | 100% | Memory hash 검증, 중복 제거 |
| Git Commit Frequency | 1+ per day | git log 자동 수집 |

### 1.2 일정 준수 지표
| 지표 | 목표 | 측정 방법 |
|-----|-----|---------|
| Design → Implementation | 3일 이내 | 설계 완료 → 구현 시작 간격 |
| Test → Deploy | 1일 이내 | 검증 완료 → 배포 간격 |
| Task Completion Rate | ≥90% | CTB에서 완료 항목 / 총 항목 |
| Blocker Resolution | 24시간 이내 | 🔴 차단 → 🟡 진행 중 시간 |

### 1.3 정보 정확성 지표
| 지표 | 목표 | 측정 방법 |
|-----|-----|---------|
| Memory Sync Accuracy | 100% | Memory와 실제 코드 상태 비교 |
| CTB Update Frequency | 매 상태 변경 시 | 커밋 시간 vs CTB 업데이트 시간 |
| User Action Clarity | 100% | 사용자 액션 항목의 명확성 검사 |
| Team Communication Lag | <30분 | 팀원 완료 → 비서 보고 시간 |

## 2. 일일 감사 프로세스 (Daily Audit Process)

### 2.1 자동 수집 (Automation)
매일 자정(00:00 KST)에 자동 실행:

```
1. Git log 수집 (지난 24시간 커밋)
2. Vercel 배포 상태 확인
3. CTB와 memory 파일 해시 비교
4. API 엔드포인트 헬스 체크 (live endpoints)
5. 팀 채널(Discord/Telegram) 메시지 수집
6. 감사 리포트 생성 → DAILY_AUDIT_REPORT.md
```

### 2.2 지표 계산
**신뢰도 점수 (Trust Score):**
```
신뢰도 = (코드품질×40% + 일정준수×30% + 정보정확성×30%)

코드품질 = (빌드성공률×50% + 테스트통과율×30% + 배포가용성×20%)
일정준수 = (설계→구현×40% + 테스트→배포×40% + Task완료율×20%)
정보정확성 = (Memory동기율×40% + CTB업데이트율×30% + 커뮤니케이션지연×30%)
```

**등급:**
- 🟢 95~100% — Excellent (개선 불필요)
- 🟡 85~94% — Good (1-2가지 개선 필요)
- 🟠 75~84% — Fair (3-4가지 개선 필요)
- 🔴 <75% — Poor (긴급 개선 필요)

## 3. 자동 개선 규칙 (Auto-Improvement Rules)

### 3.1 문제 감지 → 자동 보정
| 발견 사항 | 자동 조치 | 책임자 |
|---------|---------|-------|
| 배포 실패 | Vercel 로그 분석 → 웹개발자에게 보고 | 비서 |
| CTB 미동기 | 최신 git commit 기반으로 CTB 갱신 | 비서 |
| 빌드 경고 | SimplifyCodeAgent 실행 | 비서 |
| 메모리 중복 | 중복 항목 통합 | 비서 |
| Task 지연 | 의존성 확인 → 병렬화 가능 여부 판단 | 비서 |

### 3.2 팀원 피드백 루프
**주기:** 주 2회 (수요일, 일요일 19:00 KST)

```
1. 비서: 주간 신뢰도 리포트 생성 + 개선 제안 3가지
2. 플레너: 설계 프로세스 개선 의견
3. 웹개발자: 구현 일정 현실성 피드백
4. 평가자: 품질 기준 조정
5. 데이터분석가: 추세 분석 (신뢰도 개선율)
```

**결과:** 팀 회의 → 합의 → TEAM_IMPROVEMENT_LOG.md 기록

## 4. 신뢰도 개선 사이클

### Phase 1: 측정 (Measure) — 매일
- 자동 감사 지표 수집
- 신뢰도 점수 계산

### Phase 2: 분석 (Analyze) — 주 2회
- 문제 원인 분석
- 개선 우선순위 결정

### Phase 3: 개선 (Improve) — 즉시 또는 스케줄링
- 자동 보정 실행
- 팀 협의 필요 항목 → 회의 일정

### Phase 4: 검증 (Verify) — 다음날
- 개선 효과 측정
- 새로운 문제 확인
