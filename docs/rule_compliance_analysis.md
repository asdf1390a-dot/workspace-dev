## 🔍 근본 원인 분석 (학술/실무 자료 기반)

### 1️⃣ 상태 리포트 실패의 5가지 원인
**출처:** dfcubidesc.com "Why Status Updates Fail"

#### ❌ Too Long (너무 길다)
- 모든 작업 세부사항을 일기처럼 써서 읽지 않음
- **우리 팀:** 15:00/12:00/14:00 리포트가 5분 이상 소요?

#### ❌ Too Vague (너무 모호하다)
- "잘 진행 중" → 무엇이 진행 중? 리스크는 없나?
- **우리 팀:** 리포트가 구체적 수치 없이 정성적?

#### ❌ Too Frequent (너무 자주)
- 일일 리포트 → 일주일 단위 업무도 매일 리포트 → 노이즈화
- **우리 팀:** Asset Master는 8일 프로젝트, 일일 리포트 의미?

#### ❌ Too Late (너무 늦다)
- 문제 발생 후 보고 → 대응 불가
- **우리 팀:** 블로킹이 발생하고 나서 리포트?

#### ❌ Not Actionable (실행 가능하지 않다)
- 읽은 후 무엇을 해야 할지 불명확
- **우리 팀:** 리포트 읽은 후 누구가 어떤 조치?

### 2️⃣ 프로세스 미준수의 5가지 근본 원인
**출처:** LinkedIn - Yogesh Kumar "Why People Don't Follow Process"

#### ❌ Cognitive Overload (인지 과부하)
- 규칙이 17단계, 5시스템에 걸쳐 있음 → 기억 불가
- **우리 팀:** 규칙이 간단한가? (Phase A 4개 규칙 + 절대 규칙 4개 = 8개)

#### ❌ Misaligned Incentives (인센티브 불일치)
- 프로세스가 느리지만 KPI는 속도 → 규칙 무시
- **우리 팀:** 리포트 제출이 웹개발자/평가자/플레너의 KPI에 영향?

#### ❌ Process Isn't Embedded (도구 미통합)
- 규칙이 SOP 문서에만 있고 도구에 없음
- **우리 팀:** Discord에서 자동으로 리포트 미리알림 설정?

#### ❌ No "Why" (왜인지 설명 없음)
- "15:00에 리포트 해" but "왜?"
- **우리 팀:** 규칙의 목적이 명확한가?

#### ❌ Built Without User Input (하향식 설계)
- 리더쉽이 규칙 정한 후 실행 → 현장 한계 반영 없음
- **우리 팀:** 웹개발자/평가자/플레너에게 의견 구한 적 있나?

## 💡 개선 원칙 6가지
**출처:** LinkedIn - Yogesh Kumar & monday.com

### ✅ Make it Intuitive
→ 규칙이 쉬워야 따름

### ✅ Make it Visible
→ 도구에 통합해서 눈에 띄게

### ✅ Make it Measurable
→ 대시보드로 추적 가능하게

### ✅ Make it Useful
→ 팀원에게 직접 이득이 되도록

### ✅ Make it Flexible
→ 경직된 규칙 아니라 상황 대응 가능하게

### ✅ Make it Owned
→ 팀원들이 규칙 개선에 참여하고 책임감 가지기
