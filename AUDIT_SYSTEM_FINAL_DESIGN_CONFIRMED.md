## 📋 승인 3대 조건 (필수 구현)

### 조건 1️⃣: 즉시 알림 메커니즘
**요구사항:** DRS <85% 감지 → 1분 내 Telegram 알림 (CEO에게 직접)

**구현 방식:**
```
Vercel Cron (2분 주기) 또는 Supabase Realtime Webhook
├─ Event: DRS <85% 계산 완료
├─ Trigger: Telegram Bot API → CEO DM
├─ Template: "🔴 Critical DRS <XX%\n저장소: XX.X%\n즉시 점검 필요"
└─ Log: audit_alerts 테이블에 기록 (timestamp, drs_value, alerted_at)
```

**검증 방식 (평가자):**
- 실제 DRS <85% 발생 시 1분 내 알림 도착 시간 측정
- 점검 일시: 2026-05-21 15:00 (Day 2 회의)

---

### 조건 2️⃣: 목표 단계별 조정 (Phased DRS Targets)
**주간 목표 (엄격한 기준):**

```
Week 1-2 (05-20~26):  DRS ≥ 90% (여유 기간, 안정화)
Week 3-4 (05-27~06-02): DRS ≥ 92% (개선 단계)
Week 5-6 (06-03~09):  DRS ≥ 93% (강화)
Week 7+   (06-10~):   DRS ≥ 95% (정상 운영)

⚠️ 주의: W1~W2 중 DRS <85% 발생 시 → 추가 2주 연장
```

**책임:**
- 평가자: 주간 목표 검증 + 달성 여부 리포팅
- 웹개발자: 주간 목표 기준에 맞춘 기능 완성도 조정
- CTB 기록: ACTIVE_WORK_TRACKING.md "주간 목표 섹션"에 매주 월요일 업데이트

---

### 조건 3️⃣: 메트릭 언어 명확화
**개선 사항:**

| 개선 전 | 개선 후 | 의미 |
|---------|---------|------|
| "저장소 신뢰도 96.8%" | "백업 복구 가능률: 96.8%" | 저장된 파일 1000개 중 968개 완벽 복구 가능 |
| "Reliability Score 95%" | "일일 자동화 신뢰도: 95%" | 일일 100개 스케줄 중 95개 예약 시간 정시 완료 |

**적용 대상:**
- 일일 감사 리포트 (DAILY_AUDIT_REPORT)
- Telegram 일일 알림 메시지
- Discord #감사시스템 채널 공지
- 대시보드 표시 항목

**책임:** 데이터분석가 + 웹개발자 (메트릭 언어 재정의 + 메시지 적용)

## 🔴 3대 리스크 관리 계획

### Risk 1: 즉시 알림 지연 (SLA 1분 미충족)

**발생 시나리오:**
- Vercel Cron 네트워크 지연 > 60초
- Telegram API 응답 불안정 (peak 시간)
- 단일 채널 장애

**대응 방안:**
- [x] Vercel Cron 주기: 10분 → **2분**으로 단축
- [x] Telegram Bot 재시도 로직: 최대 3회 (5초 간격)
- [x] 대체 채널: Discord DM (즉시 알림 2중화)
- [x] 모니터링: alert_delivery_time 메트릭 매일 추적

**책임:** 웹개발자 (구현) + 평가자 (검증)  
**점검 일정:** 2026-05-21 15:00 (Day 2) → 2026-05-23 18:00 (최종 검증)

---

### Risk 2: DRS 계산 오류 (메트릭 신뢰도 훼손)

**발생 시나리오:**
- 저장소 검사 로직 부분 실패 → 낮은 DRS 발생
- "복구 불가능" 잘못 판정 (네트워크 일시 지연으로 인식)
- Supabase 쿼리 타임아웃 → 계산 중단

**대응 방안:**
- [x] 저장소 검사 재확인 메커니즘: 3회 검사 후 확정
- [x] 부분 실패 vs 완전 실패 구분 명확화
  - 부분 실패 (1회 검사 실패): retry → 통과 시 성공 인정
  - 완전 실패 (3회 모두 실패): 오류로 기록
- [x] 계산 과정 로깅: 각 파일/저장소별 검사 상태 audit_calculation_logs 테이블 기록
- [x] 평가자 수동 검증: 주 1회 월요일 09:00 (계산식 + 샘플 검사)

**책임:** 데이터분석가 (계산식) + 웹개발자 (로깅 + 재확인 로직)  
**점검 일정:** 2026-05-22 11:00 (Day 3 QA)

---

### Risk 3: 팀원 피로도 상승 (3일 압축 개발)

**발생 시나리오:**
- 웹개발자: Asset Master (30%) + Backup Phase 2 + Audit 동시 진행
- 일일 8시간 이상 집중력 저하 → 버그 증가
- 야간 연장 작업 → 품질 저하

**대응 방안:**
- [x] 병렬 분업 명확화: Asset (20%) + Audit (70%) + 휴식/여유 (10%)
- [x] 일일 15:00 팀 미팅 (30분): 진도 + 블로킹 + 피로도 체크
- [x] 야간 연장 금지: 18:00 엄격 마감 (긴급 제외)
- [x] 감시 시스템: ACTIVE_WORK_TRACKING.md 일일 진도 + 피로도 추적

**책임:** 플레너 (일일 모니터링) + CEO (리소스 조정 의사결정)  
**점검 일정:** 매일 15:00 회의 + Day 3 최종 평가
