# 팀 성능 평가 시스템

**목표**: 각 팀원(AI)의 업무 이력을 기록 → 상호 평가 → 다음 업무에 반영

---

## 📋 구조

```
team/
├── work_history/
│   ├── SECRETARY.md         (비서)
│   ├── WEB_BUILDER.md       (웹개발자)
│   ├── PLANNER.md           (플레너)
│   ├── TRANSLATOR.md        (번역가)
│   ├── DATA_ANALYST.md      (데이터분석가)
│   └── EVALUATOR.md         (평가자)
├── evaluation/
│   ├── EVALUATION_FORM.md   (평가 양식)
│   └── results/
│       └── 2026-05-14-results.md
└── TEAM_PERFORMANCE_DASHBOARD.md
```

## 📝 이력 기록 양식

### 파일명
`team/work_history/{ROLE_NAME}.md`

### 내용 구조
```markdown
# {팀원명} 업무 이력

## 2026-05-14

| 작업 | 산출물 | 상태 | 영향 |
|------|--------|------|------|
| 설계 문서 작성 | DESIGN.md (500줄) | ✅ 완료 | **예상 개발 기간 예측 불확실 → 설계 후 2.8주 (정확도 95%) 확정**; 다음 단계 개발 가능 |
| 팀원 협의 | 의견 수집 + 최적화 | ✅ 완료 | **제안 5개 → 채택 4개 (80%)** + 1개 보류; 3가지 기술 리스크 해결 |
```

**컬럼:**
- **작업**: 무엇을 했는가
- **산출물**: 파일명, 줄수, 또는 산출 형태
- **상태**: ✅ 완료 / ⏳ 진행 중 / 🟡 검토 중
- **영향**: 정량화된 다음 단계 영향 (예: 위험도 3→1, 기간 단축 20%, 채택률 80% 등)

## ⭐ 평가 양식

### 평가 항목 (5점 척도)

| 항목 | 정의 |
|------|------|
| **품질** (Quality) | 산출물의 정확성/완성도/기술적 우수성 |
| **속도** (Speed) | 예정 일정 대비 실적, 효율성 |
| **소통** (Communication) | 의견 수렴, 협업, 피드백 반영 |
| **혁신** (Innovation) | 제안 품질, 새로운 기술/방식 도입 |
| **신뢰성** (Reliability) | 지속성, 일관성, 예측 가능성 |

---

### 📐 평가 척도 상세 기준서 (Rubric)

#### 품질 (Quality)
| 점수 | 기준 | 사례 |
|------|------|------|
| **1점** | 요구사항 미충족, 오류 30% 이상 | 기술적 오류로 실행 불가, 대부분 재작업 필요 |
| **2점** | 부분 충족, 수정 필요 (20~30% 오류) | 주요 기능은 있으나 엣지케이스/예외 처리 누락 |
| **3점** | 기본 요구 충족, 미흡한 부분 (5~10%) | 정상 동작하나 최적화·완성도 개선 여지 |
| **4점** | 기대치 충족, 사소한 개선점 | 완성도 높음, 리뷰 피드백 1~2개 정도 |
| **5점** | 기대 초과, 벤치마크 수준 | 재사용 가능, 타 팀원의 학습 사례, 예상 대비 우수 |

#### 속도 (Speed)
| 점수 | 기준 | 사례 |
|------|------|------|
| **1점** | 2배 이상 지연 | 예정 2주 → 4주 이상 |
| **2점** | 20~100% 지연 | 예정 2주 → 3주 이상 |
| **3점** | 0~20% 지연 또는 기한 내 완료 | 예정 내 완료하거나 약간 초과 |
| **4점** | 기한 내 조기 완료 (10% 이상 단축) | 예정 2주 → 1.8주 |
| **5점** | 기한의 20% 이상 조기 완료 + 품질 유지 | 예정 2주 → 1.6주 이상 단축 |

#### 소통 (Communication)
| 점수 | 기준 | 사례 |
|------|------|------|
| **1점** | 일방적, 폐쇄적 | 협의 없이 진행, 피드백 무시 |
| **2점** | 최소 협의만 수행 | 주요 결정 후 보고, 제안 반영 낮음 |
| **3점** | 기본 협의 수행, 대부분 반영 | 주간 1~2회 협의, 피드백 70% 반영 |
| **4점** | 능동적 협의, 대부분 반영 (90%) | 주간 2회 이상, 의견 수렴 결과 반영 |
| **5점** | 투명하고 지속적, 상호 신뢰 | 주간 3회 이상, 피드백 100% 검토 후 채택/반박 근거 제시 |

#### 혁신 (Innovation)
| 점수 | 기준 | 사례 |
|------|------|------|
| **1점** | 기존 방식만 답습, 개선안 없음 | 이전과 동일한 기술·프로세스만 사용 |
| **2점** | 기존 방식 중심, 미미한 개선 | 기존 기술 선택, 사소한 최적화만 제안 |
| **3점** | 표준 방식 선택, 1~2가지 개선안 | 업계 표준 기술 + 1가지 신규 최적화 |
| **4점** | 새로운 기술/접근법 도입, 근거 있음 | 3가지 옵션 검토 후 가장 혁신적 선택 + 기술 PoC |
| **5점** | 창의적 해결책, 재사용 가능한 패턴 도출 | 기존에 없던 접근 방식, 팀 전체 학습 자료화 |

#### 신뢰성 (Reliability)
| 점수 | 기준 | 사례 |
|------|------|------|
| **1점** | 예측 불가, 반복되는 실수 | 같은 오류 2회 이상, 일정 불확실 |
| **2점** | 불안정, 예측 어려움 | 반복되지는 않으나 품질·일정 편차 큼 |
| **3점** | 기본 일관성, 예측 가능 | 일정 준수율 80~90%, 품질 안정적 |
| **4점** | 높은 일관성, 거의 항상 예측 가능 | 일정 준수율 95%, 품질 저하 최소 |
| **5점** | 완벽한 일관성, 항상 예측 가능 | 일정 100% 준수, 품질 편차 거의 없음 |

### ⚠️ 평가 시 주의사항

**후광 효과(Halo Effect) 방지:**
1. 각 항목을 **독립적으로** 평가 (이전 점수 보지 않고)
2. 점수 결정 전 **구체적 근거 2~3가지** 찾기
3. 최근 1~2건 업무만 평가하지 말고, **전체 주차 업무** 고려

**비현실적 극단 점수 주의:**
- 점수 5점: 문자 그대로 "재사용 가능"하거나 "팀 벤치마크" 수준이어야 함
- 점수 1점: 요구사항 미충족 또는 오류 30% 이상이어야 함
- 대부분 3~4점 범위로 수렴하는 것이 정상

### 평가 양식
```markdown
## {평가 대상 팀원명} 평가 (평가자: {평가자 이름})

### 2026-05-14 업무 평가

| 항목 | 점수 | 코멘트 |
|------|------|--------|
| 품질 | 5점 | 설계 문서가 매우 상세하고 실행 가능함. 기술 스택 선택도 합리적. |
| 속도 | 4점 | 예정보다 1일 빨리 완료. 설계 단계에 충분한 시간 투자 |
| 소통 | 5점 | 팀원들과 활발히 협의, 의견을 반영한 개선안 도출 |
| 혁신 | 4점 | polling 방식 선택이 합리적이나, 더 창의적 솔루션 탐색 가능했을까 |
| 신뢰성 | 5점 | 예정 일정 준수, 품질 저하 없음 |

**평균점수**: 4.6점

**종합 코멘트**: 
- 강점: 설계 깊이, 팀 협업, 일정 준수
- 개선점: 혁신성 강화, 위험 분석 더 깊이 있게
- 다음 반영사항: 기술 선택 시 3가지 이상 옵션 검토 후 선정
```

### 평가 대상: **활성 팀원만 (업무 있는 팀원)**

**현재 상태 (2026-05-14):**
- ✅ **평가 대상**: 비서, 웹개발자, 플레너 (3명)
- ⏸️ **평가 대기**: 번역가, 데이터분석가, 평가자 (업무 할당 후 평가 시작)

**원칙**: 
- 현주에 산출물이 있는 팀원만 평가 대상
- 비활성 팀원은 평가 데이터가 없으므로 제외
- 업무 할당 시 다음주부터 평가 포함

### 평가 제출 방식

1. **각 팀원이 직접 평가 파일 생성**
   - 경로: `team/evaluation/results/2026-05-21/`
   - 파일: `{평가대상}_by_{평가자}.md`
   - 예: `WEB_BUILDER_by_SECRETARY.md`
   - 양식: 아래 EVALUATION_FORM.md 템플릿 사용

2. **메타데이터 포함** (파일 최상단)
   ```yaml
   ---
   evaluator: SECRETARY
   target: WEB_BUILDER
   week: 2026-05-21
   scores:
     quality: 5
     speed: 4
     communication: 5
     innovation: 4
     reliability: 5
   date_submitted: 2026-05-17 18:00
   ---
   ```

3. **비서 검증** (금요일 19:00)
   - 모든 평가 점수 다 제출되었는지 확인
   - 극단적 점수 패턴 확인 (예: 항상 5점만 주는 평가자)
   - 이상값 발견 시 평가자에게 근거 재확인

4. **대시보드 업데이트**
   - `TEAM_PERFORMANCE_DASHBOARD.md`에 주간 종합 점수 반영

## 📌 원칙

- **모든 평가는 비재정적** (승급/감봉 아님) — 순전히 능력 개발 목적
- **익명성 보호** — 평가 결과는 개인 피드백, 종합 결과만 공유
- **1주 단위** — 매주 목요일 이력 정리, 금요일 평가, 월요일 반영
- **피드백 문화** — 점수보다 코멘트 중시, "왜"를 설명하는 코멘트 권장
