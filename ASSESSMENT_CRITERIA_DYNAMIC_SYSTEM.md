**목적:** 평가 기준을 고정하지 않고, 매월 외부 전문가 기준 + 기술 발전 속도를 반영하여 동적으로 업데이트  
**적용 대상:** Audit System, Asset Master, Backup App, Travel App, Portfolio Career 등 모든 프로젝트  
**운영:** 비서 주도 (팀 의견 수렴) + 월간 리뷰 회의

### Step 1: 외부 전문가 기준 수집 (08:00~10:00, 2시간)

**비서가 수행:**
1. 각 도메인별 업계 리더 기준 조사
   - **웹개발:** Vercel/Next.js 팀 블로그 + Chrome DevTools 성능 가이드
   - **QA:** Google Testing Blog + Selenium/Playwright 권장사항
   - **데이터:** Kaggle 톱 데이터사이언티스트 기준 + Google Analytics 모범사례
   - **번역:** Google Translate API 정확도 벤치마크 + 전문 번역사 표준

2. 산출물: `EXTERNAL_BENCHMARKS_YYYYMM.md`
   ```markdown
   ## 2026-05 외부 전문가 기준
   
   ### 웹개발 (Next.js 최신 기준)
   - Core Web Vitals: LCP <2.5s, CLS <0.1, FID <100ms (Google)
   - API 응답: <150ms (Vercel 권장)
   - 빌드 시간: <60s (최신 업데이트)
   
   ### QA
   - 테스트 커버리지: >85% (Google Testing Best Practices)
   - 자동화율: >80% (Selenium 기준)
   
   ### 번역
   - 정확도: >96% (Google Translate API 벤치마크)
   - 용어 일관성: 100% (전문가 표준)
   ```

### Step 2: 기술 발전 속도 분석 (10:00~14:00, 4시간)

**데이터분석가 주도:**
1. 지난 1개월 기술 변화 분석
   - 새 라이브러리/프레임워크 출시
   - 성능 개선 사항
   - 보안 업데이트
   
2. 산출물: `TECHNOLOGY_VELOCITY_YYYYMM.md`
   ```markdown
   ## 2026-05 기술 발전 속도 분석
   
   ### 영향도 High
   - React 19 출시 (Context 성능 50% 개선)
   - TypeScript 5.4 (더 빠른 타입 체킹)
   - Supabase RLS 업데이트 (쿼리 성능 30% 개선)
   
   ### 우리의 대응
   - Next.js 14.1 → 14.2 업그레이드 검토
   - API 응답 기준 <150ms로 상향 가능
   ```

### Step 3: 현재 기준의 문제점 검토 (14:00~16:00, 2시간)

**전체 팀 (Discord 또는 Telegram 회의):**

각 팀원이 1가지씩 의견 제시:
- **웹개발자:** "현재 API <200ms 기준이 너무 느림 → <150ms로 상향할 수 있음"
- **평가자:** "신뢰도 95% 목표가 너무 낮음 → 98%로 상향 필요"
- **데이터분석가:** "현재 기술로는 <100ms 충분히 가능"
- **번역가:** "정확도 기준이 없음 → 96% 추가 필요"
- **플레너:** "기준 변경 시 개발 일정 영향 검토"

산출물: `TEAM_FEEDBACK_YYYYMM.md`

### Step 4: 신규 기준 확정 + 기록 + 공지 (16:00~17:00, 1시간)

**비서 최종 작성:**

1. **신규 기준 확정:** `ASSESSMENT_CRITERIA_YYYYMM.md` (현재 월)
   ```markdown
   # 2026-05 평가 기준 (확정, 2026-05-15 16:30)
   
   ## 변경사항
   ✅ API 응답: <200ms → <150ms (기술 발전 기반)
   ✅ 신뢰도: 95% → 98% (팀 의견 기반)
   ✅ 번역 정확도: (신규) 96% 이상
   
   ## 근거
   - 외부 기준: Vercel 권장 <150ms
   - 팀 의견: 데이터분석가 "현재 기술로 충분"
   - 기술 속도: 새 Supabase RLS 쿼리 최적화
   ```

2. **기록:** MEMORY.md + 커밋
   ```bash
   git add ASSESSMENT_CRITERIA_202605.md EXTERNAL_BENCHMARKS_202605.md TEAM_FEEDBACK_202605.md
   git commit -m "chore(eval): 2026-05 평가 기준 동적 업데이트 확정
   
   외부 기준 (Vercel, Google) + 팀 의견 수렴 결과:
   - API <150ms (개선)
   - 신뢰도 98% (상향)
   - 번역 정확도 96% (신규)
   
   근거: EXTERNAL_BENCHMARKS_202605.md"
   ```

3. **공지:** 전체 팀에 Telegram 알림
   ```
   🔔 2026-05 평가 기준 업데이트 완료
   ```

## 4️⃣ CEO 마인드 강화: 기준 적용 프로세스

### 각 팀원이 평가 기준을 받은 후

**Step 1: 왜 이 기준인가 이해 (15분)**
```
예) API <150ms 기준
- 외부 기준: Vercel 최신 권장
- 우리 상황: 현재 기술로 가능
- 경쟁력: 업계 평균보다 25% 빠름
```

**Step 2: 우리가 달성할 수 있는가 검증 (30분)**
```
테스트:
1. 현재 상황: 실측 데이터 확인
2. 갭: <150ms와의 차이 분석
3. 계획: 어떻게 달성할 것인가
```

**Step 3: 주도적으로 반영하기 (지속)**
```
매일 체크:
- 새 기능 추가할 때 <150ms 고려
- 코드 리뷰에서 "이건 <150ms 기준 충족하나?"
- 배포 전 성능 프로파일링
```

## 🚀 자동 정보 수집 시스템과의 연계

이 평가 기준 시스템은 **자동 정보 수집 시스템**에 직접 피드백됩니다:

```
자동 정보 수집 에이전트
  → GitHub Trending, Product Hunt, Dev.to 수집
  → AI 필터링 (역할별 관련도)
  → 배포 (Telegram)
  
+ 평가 기준 동적 시스템
  → "최신 외부 기준이 무엇인가?"
  → "우리 기준과 차이가 뭔가?"
  → "적용 가능한가?"
  
= CEO 마인드로 작동하는 팀
```

예) 새로운 성능 프레임워크 발견
1. 자동 정보 수집: "Astro 2.0 출시, 빌드 시간 50% 개선"
2. 웹개발자: "현재 Next.js는? API <150ms 달성 가능한가?"
3. 월간 평가 기준 리뷰: "빌드 시간 기준 <30s로 상향 고려"
4. 최종 결정: 도입 또는 현 상태 유지
