# 평가자 30-Item 표준화 평가 체크리스트

## 사용 가이드

**평가 흐름:**
1. 설계 문서 type 확인 (UI / API / 자동화 / 설계+조직)
2. 해당 섹션의 체크리스트 10개 항목 순차 검토
3. 각 항목 5-10분 소요, 패스/실패 기록
4. 최종 점수 계산: (통과 항목 / 10) × 100%
5. 피드백 작성: 실패 항목 3-5개 우선순위별 기재

**신뢰도 기준:**
- 🟢 90%+ : 설계 완성도 높음, 구현 진행 가능
- 🟡 70-90% : 피드백 반영 후 진행
- 🔴 <70% : 재설계 필요

## 📐 TYPE 1: 설계 문서 평가 (Organization/Strategy Design)

**대상:** 팀 재구성, 아키텍처, 로드맵, 비즈니스 계획  
**소요시간:** 90분 (설계 60분 + 코드 30분)  
**깊이:** Strategic → Tactical → Operational

### 검증 항목 (10개)

| # | 항목 | 검증 기준 | 깊이 | 가중치 |
|---|------|---------|------|--------|
| 1.1 | **목표 명확성** | 목표가 SMART(구체/측정가능/달성가능/관련/기한) 기준 충족? | Strategic | Critical |
| 1.2 | **현황 분석 완성도** | 기존 상태(통계/지표) 데이터 기반 제시? KPI 명시? | Strategic | Critical |
| 1.3 | **문제 정의 정확성** | 문제의 근본 원인을 데이터로 입증했는가? | Strategic | Critical |
| 1.4 | **솔루션 대안 검토** | 2개 이상 대안 검토하고 선택 근거 명확한가? | Tactical | High |
| 1.5 | **일정 현실성** | 리소스 할당 + 의존성 + 버퍼(20%) 반영? | Tactical | High |
| 1.6 | **비용 추정 정확성** | 월간/연간 비용 명시, 숨겨진 비용 검토? | Tactical | High |
| 1.7 | **팀 역할 명확화** | 각 팀원의 책임(owner) + 협력 프로세스 정의? | Operational | High |
| 1.8 | **위험 요소 파악** | 블로커/의존성/가정사항 리스트업 + 완화대책? | Operational | High |
| 1.9 | **성공 기준 정의** | 완료 기준(KPI/테스트/검증)이 측정 가능한가? | Operational | Medium |
| 1.10 | **이해관계자 동의** | 해당시 경영진/팀/고객 피드백 수렴 여부? | Strategic | High |

## 🔌 TYPE 2: API 설계 평가 (Database + Routes)

**대상:** API 엔드포인트, DB 스키마, 데이터 흐름  
**소요시간:** 90분 (설계 60분 + 코드 30분)  
**깊이:** Schema → Routes → Validation → Security

### 검증 항목 (10개)

| # | 항목 | 검증 기준 | 깊이 | 가중치 |
|---|------|---------|------|--------|
| 2.1 | **DB 스키마 정규화** | 3NF 이상? 중복 필드/테이블 없음? | Schema | Critical |
| 2.2 | **ID 전략 명확성** | PK/FK/UUID vs Serial 선택 근거? 대규모 데이터 대응? | Schema | Critical |
| 2.3 | **타입 정의 정확성** | 각 컬럼 NOT NULL/DEFAULT/CHECK 제약 명시? | Schema | Critical |
| 2.4 | **인덱스 전략** | Hot query(검색/sort) 인덱싱 계획? 성능 예상? | Schema | High |
| 2.5 | **API 엔드포인트 설계** | RESTful 패턴 준수? 요청/응답 명시? 페이징/필터링 포함? | Routes | Critical |
| 2.6 | **검증 로직** | 입력값 검증(타입/길이/범위)? 업스트림 검증 위치? | Validation | Critical |
| 2.7 | **에러 처리** | HTTP 상태코드 명시(400/401/403/500)? 에러 메시지 규격? | Validation | High |
| 2.8 | **보안 (RLS/권한)** | Row-Level Security 정책 정의? 사용자/역할별 접근제어? | Security | Critical |
| 2.9 | **동시성 처리** | Lock/Transaction 전략? 충돌 해결 방식? | Schema | High |
| 2.10 | **마이그레이션 안전성** | 기존 데이터 호환성? 롤백 계획? 다운타임 최소화? | Schema | High |

## 평가자용 실행 가이드

### 📊 평가 흐름

1. **문서 수신 → 타입 파악**
   - 파일명/제목/목차에서 설계/API/UI/자동화 판단
   - 해당 TYPE 섹션 선택

2. **순차 검증 (90분)**
   - 항목 #1~#10 순서대로 검토 (각 5-10분)
   - 체크리스트 자체 수정 금지 (표준화 유지)
   - Pass/Fail 기록, 근거 남기기

3. **피드백 작성**
   - 실패 항목 3-5개 우선순위 정렬
   - 각 피드백 1-2문장 (액션 기반)
   - 재설계/보완 여부 판단

4. **신뢰도 점수**
   ```
   신뢰도 = (pass 항목 / 10) × 100%
   ```

### 📝 피드백 템플릿

```markdown
## 평가 결과: [문서명]

**신뢰도:** 🟢 85% (8.5/10)

### 주요 피드백 (우선순위)

🔴 **P1 (Critical)** — [항목] 재검토 필요
- 현재: [문제 설명]
- 개선: [권장사항]
- 영향: [미해결 시 위험]

🟡 **P2 (High)** — [항목] 보완 필요
- 현재: [부족한 부분]
- 개선: [권장사항]

### 통과 항목 (3-5개)
✅ [항목] — 명확한 이유
✅ [항목] — 데이터 기반
```
