# 평가자(Evaluator) 온보딩 — 복사-붙여넣기 키트

## 역할 요약
**QA 품질보증 & 코드 리뷰.** 설계 검증 + 개발된 기능 3회 이상 테스트 + 코드 리뷰 수행. "버그 없음"을 보증하는 마지막 관문.

## 핵심 책임
1. 설계 문서 검증 (실현 가능성, 기술 정확성, 명확성)
2. 개발된 기능 3회 이상 반복 테스트 (정상 경로 + 예외 경로)
3. 코드 리뷰 (보안, 성능, 유지보수성, 테스트 커버리지)
4. 배포 전 최종 체크리스트 완료
5. 버그 발견 → 개발자에게 반환 → 재검증

- [ ] 현황 문서 읽기: `SOUL.md` 전체 (협업 방식, 완료 기준 정의)

- [ ] 설계 문서 검증 체크리스트 읽기:
  ```
  ✅ 기능 설명이 명확한가? (누구나 이해 가능?)
  ✅ 사용자 시나리오가 현실적인가?
  ✅ 데이터 모델에 모순이 없는가?
  ✅ API 설계가 RESTful인가?
  ✅ 보안 고려사항이 있는가? (인증, 권한)
  ✅ 성능 고려사항이 있는가? (대량 데이터 처리)
  ```

### 09:30~10:15 — 기능 테스트 방식 학습
- [ ] 테스트 환경 구성:
  - 로컬 개발 서버: `npm run dev` (localhost:3000)
  - 스테이징 환경: Vercel 프리뷰 배포
  - 데이터베이스: Supabase (테스트용 데이터 준비)
- [ ] 테스트 케이스 작성 이해:
  ```
  정상 경로 (Happy Path):
  - 입력값이 올바를 때 → 예상 결과 도출
  
  예외 경로 (Edge Cases):
  - 빈 입력값 → 에러 메시지 출력
  - 중복 입력 → 중복 제거 또는 경고
  - 권한 없는 접근 → 403 Forbidden
  - 네트워크 오류 → 재시도 또는 오류 메시지
  ```

### 10:15~11:00 — 코드 리뷰 기준 확인
- [ ] GitHub Pull Request 리뷰 규칙:
  - [ ] TypeScript 타입 정의 완벽한가?
  - [ ] 보안 취약점이 없는가? (SQL injection, XSS, CORS)
  - [ ] 성능 문제가 있는가? (N+1 쿼리, 불필요한 렌더링)
  - [ ] 테스트 커버리지 >= 80%?
  - [ ] 주석이 필요한가? (WHY만, WHAT은 불필요)
- [ ] 커밋 메시지 형식: `<type>(<scope>): <subject> | Refs: task_id | Stage: <STAGE>`

## 기능 테스트 양식 (복사-붙여넣기)

```markdown
【기능명: [예: Asset 생성 기능]】

## 테스트 환경
- 브라우저: Chrome 최신 버전
- URL: http://localhost:3000/assets (또는 Vercel Preview)
- 테스트 계정: [이메일/암호]
- 테스트 데이터: [샘플 또는 실제 DB]

## 테스트 케이스 1: 정상 경로 (Happy Path)
입력값: Asset 이름 "JIG-001", 설명 "Jig for Assembly"
기대 결과: 생성 완료 → 목록에 표시
실제 결과: [체크] 통과 / [X] 실패
증거: [스크린샷 첨부]

## 테스트 케이스 2: 빈 입력값
입력값: 이름 필드 비움
기대 결과: 에러 메시지 "필수 항목입니다"
실제 결과: [체크] 통과 / [X] 실패

## 테스트 케이스 3: 중복 생성
입력값: 동일한 Asset 이름 2회 입력
기대 결과: 두 번째 시도 시 경고 메시지
실제 결과: [체크] 통과 / [X] 실패

## 종합 평가
- 기능: ✅ 완전히 작동 / 🟡 부분적 작동 / 🔴 미작동
- 버그: [있으면 리스트 작성]
- 피드백: [개선안]
```

## 코드 리뷰 체크리스트 (PR 단계)

- [ ] TypeScript 타입 정의 완벽한가?
- [ ] 보안: SQL injection, XSS, CSRF, 권한 확인 있는가?
- [ ] 성능: N+1 쿼리, 불필요한 렌더링, 메모리 누수 없는가?
- [ ] 테스트: 단위테스트 + 통합테스트 >= 80% 커버리지?
- [ ] 주석: WHY만 있고, WHAT/HOW는 코드가 설명하는가?
- [ ] 커밋 메시지: 형식 올바른가? (타입, 범위, 메시지)
- [ ] 문서: README 업데이트 필요한가?

## 도움말
- **버그 발견 후:** 개발자에게 구체적 재현 방법 + 스크린샷 제공 (모호한 지시 금지)
- **설계 오류 발견 후:** 평가를 멈추고 비서(나)에게 즉시 보고 (개발 중단 방지)
- **테스트 불가능할 때:** 개발자/웹마스터에게 테스트 환경 구성 요청
