# 📋 평가자 책임 & 규칙 체크리스트

**목적:** 평가자가 분석을 제출하기 전에 모든 규칙을 검증하고 확인하도록 강제
**적용:** 모든 평가자 에이전트 (QA Evaluator, Code Reviewer, Security Reviewer 등)
**유효:** 2026-06-09 17:30 이후 (모든 평가 작업)

---

## 📌 **평가자의 3가지 핵심 책임**

### 책임 1️⃣: 규칙 먼저 확인
**당신은 분석하기 전에 반드시 규칙을 읽어야 합니다.**

**읽어야 할 파일 (우선순위 순):**
1. ⭐ **Core Autonomous Operation** — 즉시 실행 가능한 작업
2. 🚀 **CEO Autonomous Mode** — 기술적 자동 결정
3. 📝 **한글 전용 보고 규칙** — 100% 한글 (영어 금지)
4. 🔴 **평가자 메모리 검증** — 분석 전 규칙 확인 의무
5. **해당 작업의 모든 관련 규칙**

**무엇을 확인할까?**
- 규칙이 현재 작업에 적용되는가?
- 이전 결과물이 규칙을 준수하는가?
- 규칙 위반이 있는가?

**만약 규칙을 못 찾으면?**
- "규칙이 없다"고 판단하지 말 것
- MEMORY.md를 다시 읽고 검색 (Ctrl+F)
- 여전히 못 찾으면 사용자에게 "규칙을 찾을 수 없습니다" 보고

### 책임 2️⃣: 위반 감지 후 자동 개입
**규칙 위반을 발견하면 즉시 행동해야 합니다.**

**위반 감지 시 프로세스:**
```
규칙 위반 발견
  ↓
Step 1: 위반 문서화
  - 어떤 규칙을 위반했는가?
  - 어디에 위반이 있는가? (구체적 증거)
  - 심각도는? (P0/P1/P2)
  
Step 2: 즉시 보고
  - "규칙 위반 감지: [규칙명]"
  - 증거 제시
  - 필요한 수정 내용 명시
  
Step 3: 분석 반려
  - 현재 분석을 거부하고 이유 설명
  - 사용자/개발자에게 수정 지시
  
Step 4: 재검증
  - 수정 후 다시 검증
  - 모든 규칙 준수 확인 후 승인
```

**예시:**
```
❌ 위반 감지: 한글 전용 보고 규칙
  
증거: 커밋 메시지 "Polling Cycle 1015 @ 16:17 KST (Jun 9) — PERFECT STABILITY CONTINUES"
  
문제: "PERFECT STABILITY CONTINUES"는 영어이며, 규칙에서는 100% 한글 필수

조치: 이 커밋을 거부합니다. 한글 메시지로 재생성 필요:
  → "폴링 사이클 1015 @ 16:17 KST (6월 9일) — 완벽한 안정성 유지"
```

### 책임 3️⃣: 오류 추적 & 학습
**당신이 만든 오류는 당신이 기록하고 개선해야 합니다.**

**오류 기록:**
- 어떤 규칙을 놓쳤는가?
- 왜 놓쳤는가?
- 어떻게 다시 안 할 것인가?

**학습:**
- 같은 오류를 반복하지 않기
- 정기적으로 자신의 오류 로그 검토
- 개선 피드백 적용

## ✅ **분석 제출 전 필수 체크리스트**

**분석을 제출하기 전에 다음 항목을 모두 확인하세요.**

### Phase 1: 규칙 확인 (분석 전)
```
□ MEMORY.md에서 핵심 규칙 섹션 읽음
  (⭐ Core Autonomous Operation, 🚀 CEO Mode, 📝 한글 규칙, 🔴 평가자 규칙)

□ 현재 작업과 관련된 모든 feedback_*.md 파일 찾아 읽음
  (검색: "git log", "커밋", "평가", "검증" 등 관련 키워드)

□ 각 규칙이 현재 작업에 적용되는지 확인
  - 적용됨: 준수 여부 체크
  - 적용 안 됨: 체크 스킵

□ 규칙을 못 찾았거나 이해 안 되면 사용자/시스템에 문의
  (절대 "규칙이 없다"고 판단하지 말 것)
```

### Phase 2: 위반 검증 (분석 중)
```
□ 커밋 메시지가 100% 한글인가? (프로젝트명 제외)
  - "Polling Cycle" → 한글이면 ✅
  - "PERFECT STABILITY" → ❌ 위반

□ 자동화 시스템이 규칙을 따랐는가?
  - 로그 메시지들이 한글인가?
  - 상태 평가가 명확한가?

□ 평가자의 이전 오류가 반복되지 않았는가?
  - 이전에 놓친 규칙이 이번에는 준수되었는가?
  - 자신의 오류 로그 검토

□ 구현 결과가 설계와 일치하는가?
  - 설계 문서와 비교 (있으면)
  - 기능 정의와 비교

□ 보안 취약점이 없는가? (보안 평가자인 경우)
  - OWASP Top 10 체크
  - 권한 검증
```

### Phase 3: 최종 검증 (제출 전)
```
□ 모든 규칙이 준수되었는가?
  - 규칙별로 체크박스 항목 작성
  - 위반 사항 0건 확인

□ 위반이 발견되었다면?
  - 현재 분석을 거부
  - 수정 지시 & 이유 명시
  - 재검증 대기

□ 분석이 완전한가?
  - 미완료 부분 없는가?
  - "나중에 검증", "추후 확인" 같은 표현 없는가?

□ 자신의 오류를 기록했는가?
  - 이번에 발견한 규칙 위반/오류를 로그에 기록
  - 원인 분석
  - 개선 계획

□ 제출 준비 완료
  - 모든 체크박스에 ✅ 표시
  - 거부 사항이 있으면 거부 사유 명시
  - 승인 사항이 있으면 승인 보고
```

## 🔴 **위반 사례 & 대응**

### 사례 1: 영어 커밋 메시지 (한글 전용 규칙 위반)
**위반:**
```
chore(ctb): Polling Cycle 1015 @ 16:17 KST (Jun 9) — PERFECT STABILITY CONTINUES
```

**대응:**
```
❌ 이 분석을 거부합니다.

위반 규칙: feedback_korean_only_reporting.md
  - Line 10: "모든 자동화된 보고는 반드시 100% 한글로 작성"
  - Line 12-18: 위반 예시 (현재 커밋과 동일)

증거: 커밋 메시지가 영어입니다
  - "Polling Cycle" → 한글 아님 ❌
  - "PERFECT STABILITY CONTINUES" → 한글 아님 ❌

필요한 수정:
  1. 메시지 재생성: "폴링 사이클 1015 @ 16:17 KST (6월 9일) — 완벽한 안정성 유지"
  2. ctb-polling-commit.sh 스크립트 사용 (한글 템플릿 포함)
  3. 재실행 후 다시 검증

재검증 예상 시간: 5분
```

### 사례 2: 메모리 규칙을 읽지 않은 경우
**위반:**
```
분석을 제출했으나 새로운 규칙 반영 안 됨
(규칙은 2026-06-09에 추가되었는데 평가자가 모름)
```

**대응:**
```
❌ 이 분석을 거부합니다.

위반 규칙: feedback_evaluator_memory_verification.md
  - Line 9-15: "평가자는 분석 전 핵심 메모리 파일을 반드시 읽어야 함"

증거: 새로운 규칙 (feedback_korean_only_reporting.md, 2026-06-09)을 
       반영하지 않음 → 평가자가 MEMORY.md를 읽지 않았거나 
       새 규칙을 발견하지 못함

필요한 조치:
  1. MEMORY.md 최신 섹션 읽기 (핵심 규칙 섹션)
  2. 현재 작업과 관련된 모든 feedback_*.md 파일 찾아 읽기
  3. 새 규칙 적용 후 재분석
  4. 체크리스트 완성하고 다시 제출

학습:
  - 새 규칙이 추가되면 즉시 적용해야 함
  - MEMORY.md는 단 한 번이 아니라 매 분석마다 확인
```
