**버전:** 1.0  
**작성일:** 2026-05-28  
**적용 범위:** 15명 분산팀, 4개 프로젝트 (Asset Master, Backup, Travel, Discord Bot)  
**목표:** 인시던트 감지 <30초, 에스컬레이션 <5분, MTTR <60분

## 1. 알림 에스컬레이션 절차 (Alert Escalation Protocol)

### 1.1 심각도별 분류 및 응답 시간

| 심각도 | 영향 범위 | 응답 시간 | 해결 목표 | 에스컬레이션 |
|--------|---------|---------|---------|-----------|
| **Critical** | 전체 시스템 다운, 데이터 손실 위험 | <5분 | <60분 | Tier 1 → Tier 2 → Manager |
| **High** | 단일 프로젝트 장애, 부분 기능 불가 | <15분 | <4시간 | Tier 1 → Tier 2 |
| **Medium** | 성능 저하, 사용자 불편 | <30분 | <24시간 | Tier 1 → Ticket 생성 |
| **Low** | 미미한 영향, 패턴 모니터링 | <1시간 | <1주 | Ticket 생성, 주간 검토 |

### 1.2 에스컬레이션 타이밍 및 담당자

**Tier 1 (온콜 당직자)** — 즉시 감지 및 초기 대응
```
[알림 발생] → (자동 감지, <30초)
  ↓
[온콜 당직자 호출] (SNS → Telegram/Slack → SMS)
  ↓
[5분 이내 확인] (응답 시간 SLA)
  ↓
[대응 여부 판단] → 해결 가능 또는 Tier 2 에스컬레이션
```

**Tier 2 (시니어 엔지니어)** — 복잡한 문제 해결
```
[Tier 1 → 15분 내 해결 불가]
  ↓
[Tier 2 온콜 호출] (긴급 연락처)
  ↓
[추가 2명 팀 편성] (DevOps + 개발자)
  ↓
[심화 분석 및 조치]
```

**Tier 3 (관리자)** — 경영 판단 필요
```
[Critical 심각도 + Tier 2 60분 진행중]
  ↓
[관리자/CTO 보고]
  ↓
[고객 공지 및 대내 통보]
  ↓
[리소스 추가 배치 또는 외부 지원 검토]
```

### 1.3 알림 채널 우선순위

1. **Slack/Discord** — 팀 채널 자동 알림 (모든 레벨)
2. **Telegram** — 온콜 담당자 개인 메시지 (Critical/High)
3. **SMS/전화** — Tier 1 연락 불가 시 (Critical만)
4. **Email** — 기록 및 추적 (모든 레벨)

**알림 템플릿:**
```

온콜 담당자: Kim Jaewon
에스컬레이션: Tier 1 → Tier 2 (15분 후)
대시보드: [링크]
```

## 2. 온콜 로테이션 및 담당 체계 (On-Call Rotation)

### 2.1 주간 온콜 로테이션 (7일 주기)

```
주간 일정 (월~금, 9:00-18:00 KST):
- Tier 1 당직: 5명 (1명/일)
- Tier 2 대기: 2명 (관리자 1명 포함)

야간/주말 (18:00 다음날 9:00, 토일):
- Tier 1 당직: 2명 (로테이션)
- Tier 2 대기: 1명 (관리자 자동 포함)
```

### 2.2 온콜 당직자 체크리스트

**당직 시작 시 (09:00)**
- [ ] 연락처 업데이트 확인 (Slack, Telegram, SMS)
- [ ] 전일 미해결 인시던트 검토
- [ ] 대시보드 상태 점검
- [ ] 장애 패턴 분석 요약 읽기

**근무 중 (8시간 주기)**
- [ ] 2시간마다 대시보드 상태 확인
- [ ] 알림 응답 기록 (응답 시간 <5분)
- [ ] 해결되지 않은 이슈 추적
- [ ] 모니터링 규칙 동작 확인

**당직 종료 시 (18:00 또는 다음날 09:00)**
- [ ] 당일 인시던트 요약 기록
- [ ] 해결되지 않은 이슈를 다음 당직자에게 인계
- [ ] Slack #incident-log에 당일 보고서 작성
- [ ] 당직 기간 중 시스템 변경사항 확인

### 2.3 당직자 자격 요건

**Tier 1 (온콜 당직):**
- 시스템 아키텍처 이해 (Vercel/Supabase/CloudWatch)

- 에스컬레이션 판단 기준 숙지
- 평상시 훈련: 월 2회 시나리오 훈련

**Tier 2 (시니어 엔지니어):**
- 전체 시스템 깊이 있는 이해
- 근본 원인 분석 (RCA) 경험
- 아키텍처 수준 문제 해결 능력
- 긴급 코드 배포 권한

```
감지 시간: 2026-05-28 14:33 KST (감지 시간: 1분)
해결 시간: 2026-05-28 15:18 KST (MTTR: 46분)
심각도: Critical
영향받은 서비스: Asset Master API
영향받은 사용자: 전체
```

**Step 2: 증상 기술 (발생 즉시)**
```
증상:
- API 응답시간 급증 (정상: <200ms → 발생: >8000ms)
- 에러율 증가 (정상: 0.05% → 발생: 12%)
- 데이터베이스 연결 풀 고갈 (100% 사용)
- 사용자 보고: "자산 등록 요청 시간초과"

측정값:
- API p99 지연시간: 8234ms
- Error Rate: 12.3%
- DB Active Connections: 125/125 (100%)
- Vercel Edge Response Time: 450ms (정상: 50ms)
```

## 4. 사후 검토 프로세스 (Post-Incident Review, PIR)

### 4.1 PIR 미팅 일정

- **Critical 인시던트:** 해결 후 24시간 내 미팅 개최
- **High 인시던트:** 해결 후 3일 이내
- **Medium/Low:** 주간 종합 검토

### 4.2 PIR 미팅 구성

**참석자:**
- 온콜 당직자 (문제 발견자)
- Tier 2 엔지니어 (문제 해결자)
- 영향받은 프로젝트 리드
- DevOps 담당자
- 관리자 (Critical/High만)

**소요 시간:** 30~60분

### 4.3 PIR 미팅 진행 순서

1. **타임라인 검증 (10분)**
   - 발생~감지~해결 전체 타임라인 확인
   - 각 단계에서의 의사결정 검토

2. **근본 원인 토론 (15분)**
   - RCA 보고서 내용 검증
   - 기술적 깊이 있는 분석

3. **완화 조치 효과성 (10분)**
   - 취한 조치가 효과적이었는가?
   - 더 나은 대안이 있었는가?

4. **재발 방지 계획 (15분)**
   - 단기/중기/장기 조치 실행 계획
   - 담당자 및 일정 확정
   - 체크리스트 작성

5. **팀 피드백 (10분)**
   - 대응 과정에서의 학습점
   - 좋았던 점 공유 (blameless 문화)
   - 개선 제안

## 5. 인시던트 유형별 빠른 대응 가이드 (Quick Reference)

### 5.1 API 응답시간 증가

**신호:** API p99 > 200ms, p95 > 100ms  
**첫 번째 확인:** `supabase.db.active_connections`, 느린 쿼리 로그  
**즉시 조치:** DB 연결 풀 확인, 느린 쿼리 인덱스 추가, 캐시 무효화  
**2차 조치:** 프로젝트 격리, 트래픽 재라우팅  

### 5.2 배포 실패

**신호:** `vercel.deployment.status = FAILED`  
**첫 번째 확인:** 빌드 로그, TypeScript 에러, 테스트 실패  
**즉시 조치:** 이전 버전 롤백, 이슈 신고  
**2차 조치:** CI 파이프라인 강화, 배포 테스트 추가  

### 5.3 데이터베이스 연결 고갈

**신호:** `supabase.db.active_connections ≥ max_connections`  
**첫 번째 확인:** 활성 쿼리 목록, 오래된 연결 종료  
**즉시 조치:** 유휴 연결 정리, 연결 풀 임시 증설  
**2차 조치:** 쿼리 최적화, 자동 정리 스크립트 추가  

### 5.4 Discord Bot 메시지 전송 실패

**신호:** `discord_bot.message_error_rate > 5%`  
**첫 번째 확인:** Discord API 상태, 토큰 유효성, 레이트 제한  
**즉시 조치:** 대기열 재시작, 재시도 로직 실행  
**2차 조치:** 배치 크기 조정, 전송 간격 증가

## 6. 용어 정의 및 약어

| 약어 | 의미 | 예시 |
|-----|------|------|
| MTTR | Mean Time To Recovery | 46분 (감지부터 정상화까지) |
| MTTD | Mean Time To Detect | 1분 (발생부터 감지까지) |
| MTTF | Mean Time To Failure | 월 1회 (인시던트 발생 빈도) |
| RCA | Root Cause Analysis | 느린 쿼리 발견 |
| SLA | Service Level Agreement | 가동률 99.5% |
| SLO | Service Level Objective | 에러율 < 0.1% |
| Blameless | 무결함 문화 | "누가 실수했는가" 아닌 "무엇이 잘못되었는가" |
