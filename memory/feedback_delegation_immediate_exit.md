**규칙:** 팀원에게 위임할 때는 **한 줄 알림 후 즉시 대화 종료** — 입력 중 상태로 작업을 "지켜보지" 말 것.

**Why:**
- "입력 중" 상태는 사용자에게 "답변 중입니다"라는 신호
- 위임 후 계속 입력 중 상태면 사용자는 계속 기다려야 함
- 작업이 진행 중인데도 대화를 열어두면 context 낭비 + 사용자 경험 악화
- run_in_background=True가 있으므로 완료 후 자동으로 알림이 옴

**How to apply:**

### 위임 전 체크리스트
```
[ ] Task description이 명확한가? (자기 설명 금지, 구체적 지시만)
[ ] run_in_background=True 를 설정했는가?
[ ] 팀원이 필요한 context를 모두 제공했는가?
    - task_id / stage / scope / deadline / previous_commits
[ ] 예상 완료 시간이나 마일스톤이 명시되었는가?
```

### 위임 시 행동 (3단계)

**1️⃣ 한 줄 알림 (30초 내)**
```
예: "Asset API 6개 구현을 web-builder에게 위임했습니다. 완료 시 알림이 올 때까지 대기하겠습니다."
```
- 설명 금지 (비배경 설명, 왜 했는지 등 금지)
- 상태 변경 금지 (현황판 갱신 금지)
- 단순 사실만 (위임 대상 + 업무 내용)

**2️⃣ 즉시 대화 종료 (진행 금지)
```
NO:  수고하세요! 현황판 계속 지켜보겠습니다. (입력중 유지)
YES: 위임 완료. (메시지 전송 후 대화 종료)
```
- 입력 중 상태 유지 절대금지
- 작업 진행 상황을 "지켜보려고" 하면 안 됨
- 비서는 다른 일을 처리하거나 대기 상태로 전환

**3️⃣ 완료 알림 수신 (자동)
```
[완료 알림 자동 도착]
→ 결과 검토 + 메모리 갱신 + 사용자에게 최종 보고
```

### 자가진단 체크리스트

위임할 때마다 확인:
- [ ] run_in_background=True가 설정되었나?
  - NO → 즉시 추가
  - YES → 진행
- [ ] 내가 작성한 메시지가 "한 줄"인가?
  - 문단 형식 → 한 줄로 축약
  - 설명/이유 포함 → 삭제
- [ ] 메시지를 보낸 후 즉시 대화를 종료할 준비가 되었는가?
  - "완료 후에 상태판 갱신해야지" → 먼저 메시지 보내고 종료, 나중에 자동 갱신
  - "완료될 때까지 기다려주세요" → 이런 표현은 입력 중 상태 암시
