## 규칙 (The Rule)

**작업이 예정보다 빨리 완료되면 → 다음 작업 일정을 즉시 당겨와서 시작**

### 예시
```
【계획】
- 14:00~15:00: Asset API 개발 (60분)
- 15:00~16:00: 테스트 (60분)

【실제】
- 14:00~14:45: Asset API 개발 완료 (45분, 15분 단축)
→ 【즉시 조치】15:15→14:45 테스트 시작 (15분 당김)
```

### 구체적 실행
1. **완료 신호 수신** (팀원 또는 시스템)
   - 예: "14:45 Asset API 완료"

2. **시간 델타 계산**
   - 예정: 60분
   - 실제: 45분
   - 차이: +15분 (확보)

3. **다음 작업 일정 재계산**
   - 원래: 15:00 시작 → 15분 당김
   - 새 일정: 14:45 시작

4. **CTB 즉시 갱신**
   - 완료 작업: 🟢 완료 마크 + 실제 완료시간 (14:45)
   - 다음 작업: ETA 변경 (15:00 → 14:45)

5. **다음 업무 착수**
   - 대기 없이 즉시 시작

## Why (과거 문제점)

**지적:** "예상시간이 한시간뒤인데 30분만에 끝나면 다음업무를 30분당긴일정으로진행해야지"

**문제점:**
- 작업이 빨리 끝나도 일정 시간까지 기다림 (낭비)
- 고정 시간(08:00, 14:00, 15:00, 18:00)에만 CTB 업데이트 → 실시간 동기화 안됨
- 다음 업무 지연 발생 (예정 시간보다 늦게 시작)
- 일일 효율성 저하

**개선 효과:**
- 낭비 시간 제거 → 일일 생산성 최대화
- 실시간 일정 조정 → 팀 간 의존성 자동 반영
- 예측 가능한 완료 시간 → 더 정확한 일정 계획

## How to Apply

### 조건 1: 실시간 감지 (Real-Time Detection)
**정기 체크포인트가 아닌 시점에 완료가 보고되면 즉시 처리**
- 스케줄링된 시간과 무관하게 즉시 갱신
- 예: 11:30에 완료 → 11:30에 CTB 갱신 (14:00까지 기다리지 않음)

### 조건 2: 시간 델타가 있으면 즉시 당겨오기
**언제든 예정보다 빠르면 다음 작업 당김**
- 5분 차이도 기록 (누적 효과 고려)
- 15분 차이 → 다음 작업도 15분 당김
- 일일 누적 당겨온 시간 추적

### 조건 3: CTB 갱신 필수 항목
```
| 항목 | 기록 방식 |
|------|---------|
| 완료 작업 | 상태 🟢 + 실제 완료시간 (HH:MM) |
| 예정 vs 실제 | "예정 60분 / 실제 45분" |
| 시간 델타 | "+15분 (확보)" |
| 다음 작업 새 ETA | "15:00 → 14:45" |
| 당겨온 사유 | "이전 작업 +15분" |
```

### 조건 4: 다음 작업 담당자에게 통보
**즉시 알림 형식:**
```
【일정 앞당김 알림】
- 이전 작업: Asset API (14:45 완료, 15분 단축)
- 당겨온 시간: 15분
- 새 시작 시간: 14:45 (기존 15:00에서 변경)
- 조치: 즉시 시작 가능
```

### 조건 5: 누적 추적 (일일 요약)
**일일 마지막(18:00~19:00)에 당겨온 시간 종합**
```
【2026-05-18 당겨온 일정 현황】
- Task 1: +15분 (14:45 완료)
- Task 2: +10분 (16:20 완료)
- Task 3: +5분 (17:50 완료)
---
- 일일 총 당겨온 시간: 30분
- 효과: 원래 18:30 완료 → 18:00 완료 (30분 단축)
```

## 매뉴얼 CTB 갱신 vs 자동화

### 현재 (2026-05-18 수동 운영)
비서가 **즉시 감지 + 즉시 갱신** 책임
- 팀원 완료 보고 수신 → CTB 즉시 갱신
- 실시간이므로 지연 = 규칙 위반

### 미래 (자동화 로드맵)
1. **Phase 1 (2026-05-25):** Git commit 감지 → 자동 CTB 갱신
2. **Phase 2 (2026-06-01):** 팀원 Telegram 완료 신호 → 자동 일정 재계산
3. **Phase 3 (2026-06-15):** AI 예측 기반 당겨오기 (3일 전부터)

## 예시 시나리오

### Scenario A: 정상적인 당겨오기
```
08:00 예정: Asset Master Phase 2 온보딩 시작
→ 07:50 완료 (10분 단축)
→ CTB 즉시 갱신 (07:50)
→ 다음 Task 09:00→07:50 당김
→ Web-Builder AI Agent에게 즉시 알림 (new start: 07:50)
```

### Scenario B: 누적 당겨오기 (연쇄효과)
```
Task 1: 예정 60분 → 실제 50분 (+10분)
→ Task 2: 예정 14:00→15:00 → 새 ETA 13:50→14:50
→ Task 2: 예정 60분 → 실제 55분 (+5분)
→ Task 3: 예정 15:00→16:00 → 새 ETA 14:55→15:55
→ 일일 누적: 15분 단축
```

### Scenario C: 지연 발생 (반대의 경우)
```
Task 1: 예정 60분 → 실제 75분 (-15분 손실)
→ 원인분석 (feedback_schedule_delay_handling.md 따름)
→ 개선대책 수립
→ Task 2 및 이후 모두 15분 밀림 (당길 수 없음)
→ 일일 마감 시간 자동 조정
```

## 월간 분석 (Performance Tracking)

- 월간 누적 당겨온 시간
- 가장 효율적인 작업 (평균 당겨온 시간 큼)
- 반복 지연 작업 (음수 값)
- 개선 방향 제안

**예시:**
```
【2026-05-31 월간 분석】
- 총 근무 일수: 17일
- 총 당겨온 시간: 8시간 42분

- TOP 효율 작업: Asset Master API (평균 16분 단축)
```
