# 비서 효율화 5가지 즉시 적용 규칙 (2026-05-15 16:30~)

## Rule 1️⃣: 즉시 지시 (Zero Confirmation-Waiting)

**규칙:** "발송 대기" 금지 → 즉시 발송

**Why:** 지시를 준비했는데 "발송해도 되나" 묻고 있으면 업무 지연 50% 발생

**How to apply:**
- ❌ "이 메시지를 팀원에게 보낼까요?" (확인 요청)
- ✅ "메시지 발송 완료. (내용 요약)" (결과 보고)
- 지시 준비 완료 → 즉시 발송 → 결과만 보고

**적용 예시:**
```
❌ 이전: "플레너에게 Asset Master 설계 시작 지시를 보낼까요?"
✅ 현재: "플레너에게 Asset Master 설계 시작 지시 발송 완료 (ETA 2026-05-19 18:00)"
```

## Rule 2️⃣: 구조 통일 (Unified Format)

**규칙:** 1줄 메시지로 통일

**Format:**
```
[지시] + [배경] + [산출물] + [ETA] = 1줄
```

**Why:** 이전 방식이 3-5줄이었는데, 1줄로 통일하면 준비 시간 75% 단축

**How to apply:**
- 모든 팀원 지시: 1줄 메시지
- 배경 + 지시 + 산출물 + ETA 명시
- 추가 설명은 링크로 (파일, GitHub raw 등)

**적용 예시:**
```
[지시] 플레너: Asset Master v2 Phase 2 설계 시작 | 배경: Audit 팀 논의 병렬 | 산출: DESIGN.md+API_GUIDE.md | ETA: 2026-05-19 18:00 KST

참고: memory/option1_ctb_hierarchical.md
```

## Rule 3️⃣: 병렬화 (Parallel Execution)

**규칙:** 독립 작업은 동시 발송

**Why:** 순차 발송(A→B→C 3번)과 동시 발송(ABC 1번)의 시간 차이 = 40% 단축

**How to apply:**
- 독립적인 업무는 한 번에 모두 발송
- 의존성이 없으면 무조건 병렬
- "완료 후 다음" 패턴 최소화

**적용 예시:**
```
❌ 이전: 플레너 지시 → 팀 분석 요청 → 개별 피드백 (3단계 순차)
✅ 현재: 플레너 지시 + 팀 분석 요청 + 개별 피드백 (1단계 동시)
```

## Rule 4️⃣: 상태 단순화 (Binary States)

**규칙:** 상태는 2가지만: `진행중` or `완료`

**Why:** "우선순위 대기", "블로킹 대기", "사용자 확인 대기" 등이 혼란 유발

**How to apply:**
- ❌ "우선순위 대기" → ✅ "진행중" (이미 결정됨)
- ❌ "블로킹 대기" → ✅ "진행중" (의존성 명시)
- ❌ "사용자 확인 대기" → ✅ "완료" (유저 액션 명시)
- 대기 상태는 필요시에만 "블로킹: (구체적 이유)"로 기록

**적용 예시:**
```
CTB 상태 표시:
❌ Asset Master 설계 | 상태: 우선순위 대기
✅ Asset Master 설계 | 상태: 진행중 | 블로킹: 없음 | ETA: 2026-05-19
```

---

## Rule 5️⃣: 확인 최소화 (Confirmation Elimination)

**규칙:** "발송했습니다" 후 결과 대기만

**Why:** 중간 확인 단계 제거 = 상호작용 50% 감소

**How to apply:**
- 발송/지시: 즉시 실행
- 중간 보고: 금지 (필요시에만 상태 업데이트)
- 최종 결과만 보고: "완료", "블로킹 발생", "ETΑ 변경" 등
- 진행 중 단계에서 "어떻게 하지?", "괜찮나?" 묻기 금지

**적용 예시:**
```
❌ "플레너에게 지시했는데 받았나요?" (중간 확인)
❌ "아직도 작업 중인가요?" (진행 점검)
✅ "플레너: Asset Master 설계 진행 중. ETA 2026-05-19 18:00" (상태만)
✅ (완료 시) "플레너: Asset Master 설계 완료. DESIGN.md 완성" (결과)
```
