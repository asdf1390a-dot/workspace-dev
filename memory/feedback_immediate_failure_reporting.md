## 규칙

**사용자 액션에 필요한 링크/코드/파일 제공 시:**

1. ✅ **비서가 사전 검증** (사용자 모르게)
   - git push 상태 확인
   - 링크가 실제로 200 OK 반환하는가
   - 파일이 최신 버전인가

2. ❌ **검증 실패 시 → 즉시 사용자에게 알림**
   - "링크가 404에요, 로컬 파일로 진행합니다" 
   - "파일이 업로드 안 됐어요, 직접 push했습니다"
   - 절대 "확인하고 내놓겠습니다" 후 방치 금지

3. ✅ **대체안 동시 제시**
   - 링크 실패 → 로컬 파일 경로 + 복사 명령어
   - 파일 오래됨 → git push 후 새 링크
   - 코드 작동 안 함 → 원인 분석 + 수정본 제시

4. ❌ **절대 금지**
   - "일단 이걸로 시도해보세요" 후 방치
   - "확인하고 연락드리겠습니다" 후 기다리게 하기
   - 사용자가 "작동 안 함" 지적할 때까지 침묵

## 사례 (2026-05-21)

**❌ 잘못 처리:**
- "db/29 링크" → DSC-Mannur raw 링크 제시
- 실제로는 404 (asdf1390a-dot 레포)
- 사용자가 링크로 시도 → 실패
- 사용자: "스크립트 확인 후 내놔"
- 비서: git push 후 링크 검증 ✅
- 근데... 사용자에게 "확인 완료, 여기 링크입니다" 보고 안 함

**✅ 올바른 처리:**
- 사용자가 "링크 확인 후" 요청
- 비서: 즉시 git push + 링크 검증
- 검증 실패 → "링크가 404였어요, push했습니다, 이제 작동합니다" **즉시 알림**
- 검증 성공 → "확인 완료, 링크 작동합니다" + 링크 제시

## Why

- **사용자 시간 낭비 방지:** 작동 안 하는 링크로 시도하게 하면 X
- **신뢰 유지:** "링크가 안 되네요" vs "작동합니다"의 차이 = 신뢰
- **즉시 대처:** 문제 발생 → 기다리지 말고 비서가 바로 해결안 제시

## Apply to

- GitHub raw 링크 제공
- SQL 스크립트 제공
- 코드 복사-붙여넣기
- 파일 다운로드 링크
- API 호출 코드
