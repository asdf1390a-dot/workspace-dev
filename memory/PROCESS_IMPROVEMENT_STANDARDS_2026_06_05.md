## 📋 1단계: 용어 정의 표준화 (Terminology)

### 상태 정의 (절대 불변)

모든 프로젝트는 정확히 4가지 상태 중 하나:

| 상태 | 정의 | 메모리 기록 시점 | 검증 레벨 | 예시 |
|------|------|---------------|---------|------|
| **READY_FOR_DEPLOYMENT** | 설계 완료, 배포 준비됨 | 설계 완료 후 | LEVEL 0 | UI 설계 완료, DB 스키마 설계 완료 |
| **DEPLOYED** | 실행 환경에서 프로세스 실행중 | 배포 직후 | LEVEL 1 | 포트 LISTEN, 프로세스 PID 확인 |
| **VERIFIED** | 기능 동작 확인됨 (API/DB/권한) | API 테스트 후 | LEVEL 2 | HTTP 200, 응답시간 < 200ms, 에러처리 확인 |
| **PRODUCTION_GRADE** | QA 통과 + 모든 엣지케이스 테스트 완료 | QA 승인 후 | LEVEL 3 | 경계값 테스트 통과, 보안 검증 완료, 성능 기준 충족 |

### 금지 용어 (절대 사용 금지)

```
❌ "완료" (정의 없음)
❌ "작동중" (검증 없음)
❌ "배포됨" (확인 없음)
❌ "검증됨" (증거 없음)
❌ "안정적" (측정 없음)
```

→ **반드시 위 4가지 상태명만 사용**

## 🧠 2단계: 자율 판단 명확화 (Autonomous Decision)

### 의사결정 매트릭스

**Q1: 정보가 충분한가?**

| 경우 | 판단 | 액션 |
|------|------|------|
| 정보 충분 (100%) | 자동 판단 실행 | 물어보지 말고 실행 |
| 정보 부족 (< 80%) | 최소 정보 확인 후 판단 | 1-2개 명확화 질문만 허용 |
| 불가능한 영역 | 사용자 의존 | 명확히 표시 (🔴 USER_ACTION) |

### 자율 판단 기준 (자동 실행 조건)

**자동 실행 (물어보지 않음):**
```
✅ 기술적 판단 (코드 작성, 배포, 테스트)
✅ 시스템 상태 확인 (포트, API, DB)
✅ 자동화 작업 (Cron, CI/CD)
✅ 정보 수집 (로그, 메트릭, 상태)
✅ 기존 계획 실행 (승인된 배포, 테스트)
```

**사용자 승인 필요 (명확히 표시):**
```
🔴 USER_ACTION: Supabase SQL 실행 (User만 접근)
🔴 USER_ACTION: 프로덕션 설정 변경 (민감한 데이터)
🔴 USER_ACTION: 새로운 배포 전략 승인 (정책 변경)
```

### 판단 표현 규칙

**❌ 금지:**
```
"Should I...?"
"어떻게 할까요?"
"진행할까요?"
"괜찮나요?"
```

**✅ 필수:**
```
"분석: [상황]"
```

## 📝 3단계: 메모리 업데이트 규칙 (Memory Rules)

### 메모리 업데이트 소유권

| 상태 | 담당자 | 업데이트 시점 | 필수 항목 |
|------|--------|-------------|---------|
| **READY_FOR_DEPLOYMENT** | 나 (비서) | 설계 완료 후 | 파일명, 라인수, 설계 상세 |
| **DEPLOYED** | 나 (비서) | 배포 완료 후 | 배포 환경, PID, 포트, 타임스탐프 |
| **VERIFIED** | 평가자 | 검증 완료 후 | 테스트 항목, 응답시간, 에러처리 |
| **PRODUCTION_GRADE** | 평가자 | QA 승인 후 | QA 체크리스트, 성능 지표 |

### 메모리 기록 규칙

**기록 시점 (절대 규칙):**

```
1. 파일 작성됨 → READY_FOR_DEPLOYMENT 기록
   증거: 파일명, git commit hash
   
2. 배포 완료됨 → DEPLOYED 기록
   증거: 포트 상태, PID, curl 응답
   
3. API 테스트 완료 → VERIFIED 기록
   증거: curl 결과, 응답시간, 에러처리 테스트
   
4. QA 통과 → PRODUCTION_GRADE 기록
   증거: QA 체크리스트, 테스트 케이스
```

**기록 포맷 (필수 구조):**

```markdown
### ProjectName
- **상태:** VERIFIED (LEVEL 2)
- **파일:** /path/to/file.js (298 LOC)
- **배포:** Vercel edge (https://...)
- **검증:**
  □ HTTP 200 ✓ (응답시간: 0.95ms)
  □ DB 연결 ✓ (쿼리: SELECT 1)
  □ 에러처리 ✓ (404, 401, 500 모두 확인)
- **테스트 날짜:** 2026-06-05 13:50 KST
- **증거:** [curl 결과 링크]
- **마감:** 2026-06-10 18:00 KST
```

### 메모리 검증 규칙 (자동 감시)

**매 4시간마다 실행:**
```sql
SELECT * FROM memory
WHERE status = 'VERIFIED' or status = 'DEPLOYED'
AND updated_at < NOW() - INTERVAL 4 hours
```

**불일치 발견 시:**
1. 실제 상태 재확인 (curl, ps aux 등)
2. 메모리 상태와 실제 상태 비교
3. 차이 발견 → 즉시 메모리 수정
4. 원인 분석 → EVALUATION_FRAMEWORK 업데이트

## 🎯 4단계: 평가 기준 표준화 (Evaluation Standards)

### LEVEL별 검증 항목 (최소 요구사항)

#### LEVEL 1: DEPLOYED
```
필수 검증:
□ 포트 상태 (lsof -i :포트)
□ 프로세스 PID (ps aux)
□ 응답성 (curl -I http://localhost:포트)

시간: 5분
```

#### LEVEL 2: VERIFIED
```
필수 검증:
□ HTTP 상태코드 (200/400/404/500)
□ 응답시간 (< 200ms)
□ JSON/HTML 형식 정상
□ 기본 기능 작동 (데이터 입출력)
□ 에러 처리 (NULL, 초과값)

시간: 20분
```

#### LEVEL 3: PRODUCTION_GRADE
```
필수 검증:
□ 경계값 테스트 (min/max, empty)
□ 특수문자 필터링 (SQL injection)
□ 권한 검증 (인증 필요 여부)
□ 동시성 테스트 (여러 요청)
□ 성능 기준 (평균 응답시간)
□ 에러 로깅 (모든 실패 기록)
□ 모바일/다국어 지원 확인
□ 보안 헤더 확인

시간: 1시간+
```

### 평가자 검증 체크리스트

**매주 금요일 16:00 KST 실행:**

```
□ 모든 "VERIFIED" 항목이 실제로 API 응답하는가?
□ 모든 "DEPLOYED" 항목이 포트 LISTEN 상태인가?
□ 메모리 기록이 실제 상태와 일치하는가?
□ 거짓신호가 있었는가? (지난 주)
□ 개선이 필요한 부분이 있는가?

거짓신호 발견 시:
→ 즉시 정정
→ 원인 분석
→ EVALUATION_FRAMEWORK 업데이트
```

## 🔄 5단계: CTB 시스템 개선 (CTB Overhaul)

### 현재 문제점

```
측정: 포트 LISTEN만 확인
결과: "99.2% 신뢰도" (거짓)
```

### 새로운 측정 항목 (3개)

| 항목 | 현재 | 새로 추가 | 가중치 |
|------|------|---------|--------|
| 포트 LISTEN | ✓ | 유지 | 20% |
| API 응답 | ✗ | **추가** | 40% |
| 데이터 흐름 | ✗ | **추가** | 40% |

### 새로운 CTB 계산

```
신뢰도 = (포트상태 × 0.2) + (API응답 × 0.4) + (데이터흐름 × 0.4)

포트상태: 3/3 LISTEN = 100%
API응답: 3/3 HTTP 200 = 100%
데이터흐름: 2/3 DB 연결 = 66%

신뢰도 = (100% × 0.2) + (100% × 0.4) + (66% × 0.4)
       = 20% + 40% + 26%
       = 86%
```

### CTB 측정 규칙

**매 30분마다:**
```bash
1. 포트 상태 확인 (lsof -i)
2. API 응답 테스트 (curl -s http://localhost:포트/health)
3. DB 연결 확인 (SELECT 1)
4. 응답시간 측정 (< 200ms 기준)
5. 에러율 계산 (< 1% 기준)
```

**결과 기록:**
```json
{
  "timestamp": "2026-06-05T13:50:00+0900",
  "services": [
    {
      "name": "Phase2A",
      "port_status": "LISTEN",
      "api_response": 200,
      "response_time_ms": 3,
      "db_connection": true,
      "reliability": 100
    }
  ],
  "overall_reliability": 86
}
```
