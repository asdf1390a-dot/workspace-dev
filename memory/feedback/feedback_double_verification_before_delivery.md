## 규칙

**코드/링크를 사용자에게 제공하기 전에 반드시 이중 검증 실행.**

**1️⃣ 검증 단계:**
- ✅ 파일이 실제로 존재하는가? (파일 경로/이름 확인)
- ✅ 링크가 실제로 작동하는가? (클릭하면 열리는가?)
- ✅ 내용이 올바른가? (파일 형식/구조 확인)

**2️⃣ 제공 규칙:**
- ❌ 검증 없이 제공 금지
- ❌ 추측으로 링크 생성 금지 (예: 저장소 이름 추측)
- ✅ 검증 완료 후에만 최종 링크 제공

**Why:** 
- 2026-05-20: GitHub 링크 제공 시 저장소 이름을 잘못 알고 있었음 (DSCMannur → asdf1390a-dot)
- 사용자가 404 에러를 맞음
- 사용자 입장: 코드 다운로드 규칙을 만들었는데 작동 안 함 = 규칙 위반

**How to apply:**
```
코드/링크 제공 전 체크리스트:
[ ] Git remote 확인 (git remote -v)
[ ] 파일 실제 존재 여부 (git log / file path)
[ ] 저장소 이름 확인 (사용자/계정명 정확한가?)
[ ] 링크 구성 (github raw URL 올바른가?)
[ ] 테스트 불가능하면 (시스템이 폐쇄적일 경우) 내부 검증만 완료 후 명시
```

**예시:**
```
❌ 잘못된 방법:
"https://raw.githubusercontent.com/DSCMannur/dsc-fms-portal/main/..."
→ 저장소 이름 추측 (검증 없음)

✅올바른 방법:
1. git remote -v 확인 → asdf1390a-dot/dsc-fms-portal 확인
2. git log 확인 → 파일이 main에 커밋됨 확인  
3. 링크 제공: https://raw.githubusercontent.com/asdf1390a-dot/dsc-fms-portal/main/...
```
