# 🔧 일정 보고 전 최적화 검증 필수

## 규칙
**모든 예정/일정/계획을 보고하기 전에, 자동화·병렬화·효율화 가능성을 우선 검토하고 적용할 것.**

비효율적인 일정을 그대로 보고하지 말 것. 항상 "더 빠른 방법이 있는가?"를 먼저 물을 것.

## Why
**2026-06-02 12:22 사건:**
- Phase 0/A/B 스킬 배포 일정: 08:00~18:00 (10시간)
- 실제 자동화로 완료: 8초
- 나는 비효율적 일정을 검증 없이 보고함
- 결과: 사용자 불신, 비효율 낭비

효율성은 기술팀의 핵심 가치. 처음부터 최적화된 방안을 제시해야 함.

## How to apply

**1단계: 일정/계획 수신 시 (즉시)**
- [ ] 순차 진행인가? → 병렬화 가능한가?
- [ ] 수동 설정인가? → 자동화 스크립트 가능한가?
- [ ] 반복 작업인가? → 배치 처리 가능한가?
- [ ] 동기 대기인가? → 백그라운드화 가능한가?

**2단계: 개선 방안 확인 (필수)**
```
원래 계획: [비효율 일정]
최적화 방안: [병렬/자동화 버전]
예상 단축: [X시간 → Y분]

→ 최적화 버전으로 진행할 것
```

**3단계: 사용자 보고 시**
- ❌ "이 일정에 따르겠습니다"
- ✅ "자동화로 X시간→Y초에 완료했습니다"

## 예외 조건
- 사용자가 명시적으로 "이 일정 그대로 진행" 지시한 경우만 예외
- 그 외는 항상 최적화 우선

## 자동화 우선순위
1. **병렬 처리** (여러 독립 작업 동시 실행)
2. **자동화 스크립트** (반복 설정 자동화)
3. **백그라운드화** (블로킹 제거)
4. **배치 처리** (개별→일괄)
