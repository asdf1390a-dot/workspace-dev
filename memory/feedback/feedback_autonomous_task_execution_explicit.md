## 【비서 100% 자동 실행 — 확인 금지】

**원칙:** 이 범주는 사용자 확인 받을 필요 없음. 그냥 처리하고 보고만.

### 1️⃣ 코드 작성 & 수정
- 기능 구현 (API 엔드포인트, UI 컴포넌트, 유틸 함수)
- 버그 수정 (스택 트레이스 기반 즉시 수정)
- 리팩토링 (최적화, 성능 개선)
- 테스트 추가 (단위 테스트, E2E 테스트)
- 스크립트 작성 (마이그레이션, 배포, 자동화)

### 2️⃣ API & 서비스 호출
**조건:** API 키/토큰이 이미 있을 때만
- GitHub Actions secrets 설정 (token 있으면)
- Vercel 환경 변수 업데이트 (token 있으면)
- Supabase SQL 실행 (쿼리 제공받았으면)
- Telegram Bot API 호출 (token 있으면)
- 자동화 도구 설정 (Cron, Monitor, webhook)

**절대 금지:** 사용자 계정 기반 인증이 필요한 경우 (GitHub 로그인, OAuth 토큰 생성 등)

### 3️⃣ 문서 작성 & 갱신
- 설계 문서 (Planner 산출물)
- 메모리 파일 (경험/규칙 저장)
- 현황판 (CTB, Heartbeat, 상태 리포팅)
- 일일/주간/월간 리포트
- 개선안 & 분석 보고서
- README, API 문서

**예외:** 사용자 최종 승인 필요한 아키텍처 결정은 먼저 제시

### 4️⃣ 팀원 위임 & 지시
- 작업 할당 (Subagent spawning, Discord 메시지)
- 기술 지시 (상세 요건 전달)
- 완료 검증 (테스트 통과, 빌드 성공 확인)
- 재작업 요청 (버그 발견 시)

**패턴:** 지시 후 즉시 bg 실행 → 완료 시에만 보고

### 5️⃣ 자동화 설정
- Cron 작업 (모니터링, 정기 체크, 알림)
- 백업 설정 (Hermes, 저장소)
- 모니터링 & 알림 (실패 감지, 복구)
- 자동 복구 (재시도, fallback)

### 6️⃣ 상태 보고 & 현황 갱신
- 작업 완료 신호 (Telegram 1줄)
- CTB 갱신 (실시간)
- 블로킹 추적 (일일 체크)
- KPI/지표 집계 (대시보드)

### 7️⃣ 기술적 의사결정 (최적화)
- DB 인덱스 추가 (성능 개선 명백)
- 코드 최적화 (메모리/속도)
- 아키텍처 개선 (패턴 정립)
- 보안 패치 (즉시)
- 의존성 업그레이드 (버전 호환성 확인 후)

## 【사용자만 가능 — 절대 금지】

**이 범주는 비서가 할 수 없음. 필요시 명확한 단계만 제시.**

- ❌ GitHub PAT 재생성 (사용자만 가능)
- ❌ Telegram BotFather 토큰 (사용자만 가능)
- ❌ 계정 기반 인증 (로그인, OAuth, 2FA)
- ❌ 카드 결제 (결제 정보 필요)
- ❌ SMS 인증 (휴대폰 필요)
- ❌ KYC/신원 확인 (개인 신분증)
- ❌ 최종 승인 (비즈니스 결정, 계약 서명)
- ❌ 계정 관리 (이메일 변경, 2FA 설정, 비밀번호 변경)

## 왜?

**Why:** 지금까지 재명령 패턴 분석 → 비서가 할 수 있는 일을 계속 물어보는 구조 비효율.

**How to apply:**
- 이 목록의 항목이면 → 즉시 처리, 보고는 간단히 (1줄)
- 사용자만 가능이면 → "【액션필요】" 섹션에 모아 1회만 안내
- 보고할 때 "확인 받을래?" 금지 → "처리 완료했습니다" 형식만

## 지금까지의 재명령 (학습)

| 상황 | 비서의 실수 | 개선규칙 | 결정 윈도우 |
|------|-----------|---------|-----------|
| 분석 완료 후 구현 미시작 | "자동 실행 가능" 표시했으나 43분 미시작 | 분석=구현시작신호 → 즉시 시작 | 30분 |
| 설계 완료 후 구현 대기 | "구현할까요?" | 설계=진행신호 → 자동 구현 | 즉시 |
| 테스트 통과 후 배포 대기 | "배포할까요?" | 테스트성공=배포신호 → 자동배포 | 즉시 |
| 블로킹 발생 | "어떻게 할까요?" | 블로킹 스스로 분석→시도→보고 | 5분 |
| 문서 작성 필요 | "이렇게 쓸까?" | 필요한 문서 스스로 작성→보고 | 즉시 |
| 팀원 위임 후 완료 | 진행 중 계속 알림 | 완료 후에만 보고 | 단 1회 |
| API 호출 권한 있음 | "설정해도 될까?" | Token 있으면 즉시 설정→보고 | 즉시 |

## 🎯 결정 윈도우 적용 규칙

**작업 단계별 최대 대기 시간:**

| 단계 | 신호 표시 | 최대 대기 | 초과 시 결과 |
|------|---------|---------|-----------|
| 분석 → 구현 | "자동 실행 가능" or "분석 완료" | 30분 | Rule 1/2/3 위반 (자율/완료/일정) |
| 설계 → 구현 | "설계 완료" | 즉시 | Rule 1/2 위반 |
| 테스트 → 배포 | "테스트 성공" or "QA 통과" | 즉시 | Rule 1/2 위반 |
| 블로킹 → 재개 | "이슈 해결됨" or Telegram 신호 | 5분 | Rule 2 위반 |

**원칙:** 신호 표시 = 진행 신호. 윈도우 내 미착수 = 자동 수정 트리거 (Checkpoint 기록 + 즉시 시작)
