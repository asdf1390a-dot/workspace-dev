## 규칙 (The Rule)

**계획보다 1분이라도 지연되면:**
1. 즉시 인지 (예정 시간 경과)
2. 원인분석 (구체적 이유 파악)
3. 개선대책 수립 (재발 방지)
4. 사용자에게 보고 (지연 + 원인 + 대책)
5. 메모리 기록 (누적 추적)

## Why

**과거 문제점:**
- "일정을 지키겠다고 했는데 자주 놓침"
- "같은 이유로 반복해서 지연"
- "지연 이유를 모름 → 개선 불가"

**개선 효과:**
- 1분 지연도 감지 → 빠른 대응 가능
- 원인 분석으로 근본 해결
- 지속적 개선으로 신뢰도 상승

## How to Apply

### 1단계: 지연 인지 (Detection)

**예정된 시간들:**
- 🕘 08:00 KST — 어제 블로킹 + 오늘 예상 확인
- 🕘 14:00 KST — 플레너 리포트 수신 후 반영
- 🕘 15:00 KST — 웹개발자 리포트 수신 후 반영
- 🕘 18:00 KST — 일일 최종 검증
- 📌 팀원 작업 완료 예정 시간 (CTB에 명시)
- 📌 사용자 액션 응답 기한

**감지 방식:**
- 예정 시간 ± 0초 → 즉시 확인
- 1분 경과 → "지연" 판정
- 사용자 메시지 회신 필요 시 → "지연" 판정

### 2단계: 원인분석 (Root Cause Analysis)

**분석해야 할 항목:**

❌ 나쁜 예:
- "바빴습니다"
- "복잡했습니다"
- "예상과 달랐습니다"

✅ 좋은 예:
- "Supabase API 응답 지연 (2분 예상) → 최적화 필요"
- "웹개발자 피드백 대기 중 (의존성) → 병렬화 방안 검토"
- "예상 예정 시간 부족 (60분 예정 → 실제 75분 소요) → 범위 재평가"
- "예기치 못한 버그 (OAuth 충돌) → 설계 단계에서 미검토"

**원인 분류:**
1. **기술 차단** (Tech Blocker)
   - 예: 배포 실패, 빌드 에러, API 응답 지연
   - 대책: 기술적 해결 + 완화 조치
   
2. **의존성 차단** (Dependency)
   - 예: 다른 팀원 완료 대기, 사용자 승인 대기
   - 대책: 병렬화, 우선순위 조정, 커뮤니케이션 강화
   
3. **예상 오차** (Estimation Error)
   - 예: 계획 60분 → 실제 75분
   - 대책: 예정 시간 버퍼 추가, 범위 축소, 프로세스 최적화
   
4. **설계 미흡** (Design Gap)
   - 예: 설계 단계에서 미고려한 복잡도
   - 대책: 설계 리뷰 강화, 체크리스트 추가

### 3단계: 개선대책 수립 (Improvement Plan)

**대책의 원칙:**
- 구체적이어야 함 (모호하지 않음)
- 반복 가능해야 함 (한 번만이 아님)
- 측정 가능해야 함 (효과 확인 가능)

**예시:**

| 지연 이유 | 개선대책 | 측정 방법 |
|---------|---------|---------|
| Supabase API 응답 지연 | 인덱스 추가 + 쿼리 최적화 | 쿼리 응답 시간 < 200ms 확인 |
| 예정 시간 부족 (60→75분) | 향후 예정 시간 +30% 추가 | 다음 유사 작업에서 예정 시간 충족 확인 |
| 웹개발자 의존성 | 병렬화 계획 수립 (미리 협의) | 다음 작업부터 병렬 진행 실행 |
| 설계 미흡 | 설계 체크리스트에 항목 추가 | 다음 설계 때부터 신규 항목 검토 |

### 4단계: 사용자에게 보고 (Report to User)

**보고 형식:**

```
【지연 보고】
- 예정: HH:MM
- 실제: HH:MM
- 지연: NN분

【원인】
[구체적 이유]

【개선대책】
[반복 방지 방안]

【다음 조치】
[이후 진행 계획]
```

**예시:**
```
【지연 보고】
- 예정: 15:00
- 실제: 15:12
- 지연: 12분

【원인】
Supabase API 응답 지연 (avg 2.5초) → 쿼리 최적화 미흡

【개선대책】
1. assets 테이블 인덱스 추가 (asset_code, status)
2. CTB 조회 쿼리 캐싱 도입
3. 향후 CTB 갱신 예정 시간 ±5분 버퍼 추가

【다음 조치】
- 금일 18:00 전 인덱스 적용 완료
- 내일부터 적용된 쿼리 성능 모니터링
```

### 5단계: 메모리 기록 (Memory Tracking)

**기록할 항목:**
- 날짜/시간
- 업무 항목
- 지연 시간
- 원인 분류
- 적용된 대책
- 효과 확인 (1주 후)

**기록 위치:**
- `active_work_tracking.md` — 당일 지연 기록
- 월간 분석 리포트 — 패턴 분석 (상위 3개 원인)

**기록 예시:**
```
### 2026-05-18 지연 기록

**15:00 CTB 갱신 → 지연 12분**
- 원인: Supabase 쿼리 지연
- 대책: 인덱스 추가 + 캐싱
- 상태: 진행 중

**18:00 평가 리포트 → 지연 5분**
- 원인: 예정 시간 부족
- 대책: 다음 평가는 +10분 할당
- 상태: 적용됨

**월간 분석 (2026-05-01~31)**
- TOP 원인: 예정 시간 부족 (45%)
- TOP 원인: API 지연 (30%)
- TOP 원인: 의존성 대기 (25%)

**개선 실적:**
- 5월: 평균 지연 8.2분
- 6월 목표: 평균 지연 < 3분
```

## 비서의 책임 (Secretary's Role)

**매일:**
- 08:00, 14:00, 15:00, 18:00 정기 체크인 시간 엄격히 준수
- CTB에 기록된 모든 팀원 완료 예정 시간 모니터링
- 1분이라도 지연 감지 시 즉시 대응

**지연 시:**
1. 즉시 원인 파악 (팀원 또는 시스템 확인)
2. 원인 분류 (4가지 중 하나)
3. 개선대책 수립 (구체적)
4. 사용자에게 보고 (형식 준수)
5. 메모리 기록 (누적 추적)

**주 1회:**
- CTB 지연 기록 검토
- 반복되는 원인 패턴 파악
- 프로세스 개선안 제안

**월 1회:**
- 월간 지연 분석 리포트 작성
- TOP 3 원인 분석
- 다음 월 개선 목표 설정
