## 규칙

규칙 위반 또는 내가 한 번 시킨 작업에서 오류가 발생하여 반복 시행이 필요한 경우, **평가자가 자동으로 개입**하여 오류 발생 이유를 확인하고 개선조치를 진행한다.

## 왜 (Why)

- 비서 AI가 같은 실수를 반복하면 신뢰도 저하
- 규칙 위반(예: 상태 거짓 기록, 설계 ≠ 배포 구분 실패)은 즉시 교정 필요
- 평가자의 독립적 감시로 자체 검증 강화

## 어떻게 적용 (How to apply)

**자동 트리거 조건:**
1. 비서가 상태를 기록했으나 검증 불통 발생
2. 사용자가 같은 작업을 2회 이상 시행하도록 요청
3. HEARTBEAT에서 규칙 위반 감지 (예: "배포 완료"라 했으나 SQL 실제 미실행)

**평가자 자동 개입 프로세스:**
1. ⏸️ 현재 작업 일시중지
2. 🔍 오류 분석 (root cause analysis)
   - 코드/설계 오류 vs 프로세스 오류 vs 커뮤니케이션 오류 구분
   - 비서의 실수 원인 규명
3. 📝 개선안 수립 및 메모리 기록
4. ⚡ 즉시 재실행 (사용자 확인 불필요)
5. 📊 보고 (한 줄 요약: "평가자 감지 — {오류} 원인: {원인} / 해결: {조치}")

**메모리 업데이트:**
- 오류 분류: `feedback_error_classification.md` (반복 패턴 추적)
- 개선 기록: `memory/logs/evaluator-auto-fix-{날짜}.log`

**주의:**
- 평가자 개입은 **자동**이며, 사용자 승인 불필요
- 사용자 지시와 규칙 위반은 다름 (사용자 지시는 따르되, 규칙 위반만 평가자 개입)
- 예: 사용자가 "일단 미완성 배포해"라 하면 → 따름 (자율 결정 아님) / 하지만 그 결과 상태를 거짓으로 기록하면 → 평가자 개입
