## 🔴 체크1: Glossary 3층 일관성 검증 (Critical)

팀 조율의 기반은 용어 통일. 번역·DB·UI 세 레이어에서 같은 용어를 쓰는지 확인.

### 1-1. 번역 기준 문서 확인

```
[ ] 최근 3개월 번역 기록에서 "failure_code" 유사 용어 혼용 현황?
    예: "모터 소손" + "Motor burnt out" + "Motor failure" 혼재 여부
    
[ ] 긴급도 표현 희석 여부?
    한국: "즉시 조치" → 영어: "Please consider addressing" (톤이 부드러워졌나?)
    
[ ] asset_id, priority 등 핵심 필드 번역 일관성?
    동일 용어가 항상 같은 영어로 번역되는가?
```

**기준:**
- ❌ 같은 용어의 3가지 이상 다른 번역 발견 → 🔴 Critical (번역 가이드 재확인 필요)
- ⚠️ 2가지 다른 번역 발견 → 🟡 Medium (용어 통일 회의)
- ✅ 일관성 있음 → 🟢 Pass

---

### 1-2. DB 스키마 `glossary` 테이블 검증

```sql
-- glossary 테이블이 모든 드롭다운 필드 커버하는가?
SELECT COUNT(*) FROM glossary WHERE field_key IN (
  'failure_code', 'event_type', 'priority', 'asset_status'
);
-- 결과: 10+ 행 이상이면 정상

-- 각 필드의 옵션 수 일관성?
SELECT field_key, COUNT(*) as option_count FROM glossary GROUP BY field_key ORDER BY option_count;
-- 한 필드가 30개 이상 옵션이면 드롭다운 복잡도 높음 (UX 재검토 필요)
```

**체크리스트:**
- [ ] `failure_code`, `event_type`, `priority` 최소 3개 필드 커버?
- [ ] 각 필드마다 20개 이하 옵션?
- [ ] NULL 값이나 중복 code 없음?
- [ ] `label_ko`, `label_en` 모두 채워짐?

**기준:**
- ❌ 필드 누락 또는 옵션 50+ → 🔴 Critical
- ⚠️ 옵션 부분 누락 → 🟡 Medium
- ✅ 정상 → 🟢 Pass

---

### 1-3. UI 드롭다운 실제 렌더링 확인

```
[ ] 웹앱 BM 등록 폼 열어서 드롭다운 옵션 확인
    - failure_code 드롭다운의 옵션이 glossary 테이블과 정확히 일치?
    - label_ko 값이 제대로 표시되나?
    
[ ] 모바일 화면에서도 드롭다운 정상?
    - 옵션 리스트가 스크롤 없이 한 화면에 보이나?
    - 터치 영역 크기는 충분한가? (최소 44px)
```

**기준:**
- ❌ 드롭다운 옵션이 glossary와 불일치 → 🔴 Critical (즉시 DB 연동 확인)
- ⚠️ 모바일 UX 문제 (스크롤 필요, 터치 영역 작음) → 🟡 Medium
- ✅ 세 레이어 일관 → 🟢 Pass

## 🟡 체크2: BM 입력 데이터 품질 Spot Check (월 1회)

번역·분석·QA가 모두 지적한 "입력 단계 표준화" 문제를 모니터링.

### 2-1. 최근 1개월 BM 기록 10건 무작위 샘플

```sql
-- 최근 1개월 BM 기록 무작위 10건 추출
SELECT * FROM bm_events 
WHERE created_at > NOW() - interval '30 days'
ORDER BY RANDOM() LIMIT 10;
```

**각 샘플에 대해 확인:**

```
1. Asset ID 드롭다운 선택 정상?
   [ ] DCMI-* 형식만 기록됨 (자유 입력 없음)
   [ ] 존재하지 않는 asset_id 값 없음

2. failure_code 표준화 정도
   [ ] 모두 glossary 테이블의 code 값과 일치?
   [ ] "모터 소손" 같은 한글 자유입력 없음?

3. priority 필드 활용 현황
   [ ] "Urgent" 선택 기준이 합리적인가?
   [ ] priority 미입력 (NULL) 건수?

4. 타임스탬프 신뢰도
   [ ] started_at 모두 채워짐? (NULL 없음)
   [ ] resolved_at: 완료 건만 입력됨?
   [ ] started_at > resolved_at 역순 오류 없음?

5. 입력 완성도 지표
   [ ] 필수 필드 모두 채워짐 (asset_id, failure_code, priority)?
   [ ] 선택 필드(notes 등)의 입력률?
```

**결과 기록:**
- ✅ 10/10 표준 준수 → 🟢 Pass (현장 입력 품질 양호)
- 🟡 7-9/10 준수 → 🟡 Medium (현장 가이드 강화 필요)
- 🔴 6/10 미만 준수 → 🔴 Critical (입력 폼 설계 재검토)

---

### 2-2. 데이터 오염 지표 모니터링

```sql
-- 비표준 failure_code (glossary 미매핑)
SELECT DISTINCT failure_code 
FROM bm_events 
WHERE created_at > NOW() - interval '30 days'
  AND failure_code NOT IN (SELECT code FROM glossary WHERE field_key = 'failure_code');
  
-- 결과 행수 = 오염 건수. 0이면 정상, 1 이상이면 🟡
```

**해석:**
- 0건 → 🟢 Pass (입력 제약 정상 작동)
- 1-3건 → 🟡 Medium (오염 원인 분석 필요 - 폼 버그? 직원 실수?)
- 4건 이상 → 🔴 Critical (드롭다운 강제 미작동, 웹개발자 즉시 확인 필요)

## 🎯 체크3: 팀 조율 체크포인트 (진행 중 프로젝트별)

각 프로젝트 진행 단계별 "구현 완료 ≠ 품질 보증" 게이트 확인.

### 3-1. 배포된 기능별 검증 게이트 진행 상황

```
기능                  | 배포 날짜 | 기대 검증 날짜 | 검증 상태 | 담당자
------------------|---------|-------------|--------|------
failure_code 드롭다운  | 2026-05-15 | 2026-06-15 | [ ] 예정 | 평가자
BM 우선순위 배지      | 2026-05-20 | 2026-06-20 | [ ] 예정 | 평가자
부품 재고 자동 차감    | 2026-05-25 | 2026-06-25 | [ ] 예정 | 평가자
MTBF 대시보드       | 2026-06-01 | 2026-07-01 | [ ] 예정 | 데이터분석가
```

**진행 중인 프로젝트별 체크 (매월 갱신):**

- [ ] 각 기능의 "배포 1개월 후 spot check" 일정 확인됨?
- [ ] QA 검증 게이트 담당자가 명시적으로 지정됨?
- [ ] 검증 기준(10건 샘플 기준 7/10 이상 합격)이 팀 간 합의됨?

**기준:**
- ❌ 검증 게이트 미수립 (배포만 했고 검증 일정 없음) → 🔴 Critical (비서가 평가자에게 즉시 확인 요청)
- ⚠️ 검증 기준 모호함 → 🟡 Medium (기준 명확화 회의)
- ✅ 일정·담당자·기준 명시 → 🟢 Pass

## 🔄 체크4: 팀 간 우선순위 순서 정렬 (Quarterly 체크, 월간 유지관리)

**비서의 핵심 역할: 각 팀이 같은 방향을 보고 있는가?**

### 4-1. 진행 중인 업무의 순서 일관성

```
현황: 여러 팀이 제시한 우선순위가 겹치거나 충돌하지 않는가?

예시 확인 항목:
- [ ] 웹개발자가 "BM 폼 우선"이라고 하고, 플레너도 같은 판단?
- [ ] 데이터분석가의 "glossary 정제 선행" 의견을 웹개발자가 수용함?
- [ ] 번역가의 "용어 기준 미합의" 지적이 폼 드롭다운 스킬 목록에 반영됨?
```

**체크 질문:**
- [ ] 지난 1개월간 우선순위 변경(스코프 크리프)이 3회 이상? → 🟡 (조율 회의 필요)
- [ ] 각 팀의 담당 작업이 서로 대기 중인 상태? → 🔴 (병목 분석 필요)
- [ ] 합의된 순서대로 진행 중? → 🟢 (정상)

## 📋 체크5: 에스컬레이션 기준 정기 재검토

팀 조율 시 "이게 왜 이상한가"를 판단하는 기준 유지.

### 5-1. 현재 에스컬레이션 기준표

| 현상 | 판단 기준 | 담당자 | 액션 |
|------|---------|--------|------|
| **긴급도 희석** | KR "즉시"→EN "please consider" | 번역가 | 톤 확인 |
| **용어 혼용** | 같은 개념 3개 이상 다른 표현 | 비서 | glossary 재검토 |
| **입력 오염** | glossary 미매핑 4건 이상/월 | 웹개발자 | 드롭다운 강제 재확인 |
| **배포 후 검증 누락** | 기능 배포 후 1개월 내 QA 기록 없음 | 평가자 | 검증 일정 수립 |
| **팀 간 의존성 충돌** | A 팀의 "B 완료 대기" + B 팀의 "A 우선" | 비서 | 우선순위 회의 소집 |

**매월 검토:**
- [ ] 위 기준이 여전히 유효한가? (현장 상황 변화 반영 필요?)
- [ ] 새로운 에스컬레이션 패턴 발견됨? (추가 기준 필요?)
