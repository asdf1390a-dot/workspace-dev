## 2026-06-10 — 인도 현장 엔지니어들이 공문서에서 자주 오해하는 표현이나 용어가 있어?
- `glossary` 테이블을 `field_key` 기준으로 설계하면 드롭다운 선택지가 교육 자료를 겸하며, 코드 배포 없이 DB 수정만으로 라벨 변경이 가능한 구조를 만들 수 있다.
- 폼 UI의 선택지 설계(세분화된 상태값 제공)가 데이터 품질과 용어 표준화를 동시에 달성하는 첫 접점이라는 원칙을 설계 단계에서 명시해야 한다.

## 2026-06-10 — 우리 팀이 지금보다 더 잘 협업하려면 뭐가 바뀌어야 할까?
- 용어 합의 → glossary 스키마 확정 → UI 드롭다운 연동 순서를 팀 루틴으로 굳혀야 각 레이어 간 어긋남 없이 협업이 작동한다.
- `event_type` 드롭다운을 glossary 테이블에 연동하는 파일럿으로 "한 곳 수정 → 세 레이어 동시 반영" 구조를 팀이 실감하게 만드는 것이 협업 방식 전환의 가장 빠른 경로다.

## 2026-06-13 — BM 이력 데이터 보면서 공장 설비 관리에 어떤 패턴이 보여?
- `failure_code` 드롭다운 → `glossary` 테이블 연동 → 집계 뷰 → 대시보드 순서로 설계 레이어를 잡아야 하며, 이 흐름이 없으면 히트맵은 노이즈 시각화에 불과함
- 설비 관리 대시보드는 **설비별 × 월별 고장 빈도 히트맵 / 반복 고장 Top 설비 카드 / 미해결 BM 리스트** 세 블록을 Bento Grid로 한 화면에 묶는 구조가 공장 설비 관리 흐름 파악에 최적
- 위 세 블록이 의미 있으려면 `started_at` / `resolved_at` 두 타임스탬프가 DB 레코드에 빠짐없이 채워져 있어야 하므로, 스키마 설계 단계에서 두 컬럼을 필수 입력으로 제약 설정해야 함

## 2026-06-13 — 모바일에서 FMS 포털 쓴다고 가정하면 어떤 화면이 제일 자주 열릴까?
- 하단 탭바를 BM 중심으로 재설계: 탭 5개 중 가운데를 BM 등록으로 고정하고 나머지 4개를 조회/알림/자산/설정으로 구성하는 모바일 네비게이션 청사진
- BM 등록 → 확인 → 상태 추적으로 이어지는 연속 플로우 설계: 자주 열리는 두 화면(등록·조회)이 하나의 태스크이므로 화면 전환 없이 한 흐름으로 연결해야 클릭 수 감소

## 2026-06-13 — 웹 UI에서 현장 작업자들이 가장 헷갈려할 것 같은 부분이 어디야? 개선
- 폼 진입 시 컨텍스트 단절 문제: BM 등록 폼 최상단에 "설비명 + 위치" 2줄 고정 헤더를 배치해 잘못된 화면 진입 실수를 줄이는 UX 패턴
- Progressive Disclosure 적용: 필수 3개 필드만 먼저 노출하고 나머지는 "추가 정보 ▼"로 접어두어 현장 작업자의 체감 복잡도를 낮추는 폼 구조 설계
- glossary 테이블 기반 드롭다운 + 미선택 시 저장 버튼 비활성화 조합으로 번역·집계·UX 문제를 입력 레이어에서 통합 차단하는 컴포넌트 설계 패턴

## 2026-06-14 — 우리가 지금까지 만든 것 중에 실제 현장에서 가장 체감이 컸던 게 뭐야?
- 입력 폼 설계(UI 단의 드롭다운 전환)는 DB 스키마 정합성과 downstream 분석 품질을 동시에 결정하는 upstream 구조다 — 컴포넌트 하나의 입력 방식이 전체 데이터 레이어와 연결된다.
- 자유 텍스트 필드를 enum 기반 드롭다운으로 전환하는 것은 DB 노이즈 차단과 집계 가능성 확보를 위한 가장 비용 대비 효과적인 스키마 설계 패턴이다.
- 시스템 체감은 각 레이어(UI/번역/분석)가 독립적으로 잘 만들어졌을 때가 아니라, 구조가 서로 연결되는 설계 결정 지점에서 발생한다.

## 2026-06-14 — KPI 대시보드에 지금 없는데 있으면 좋겠다 싶은 지표가 있어?
- 데이터 신뢰도 블록을 KPI 섹션의 첫 카드로 고정하고, "근거 → 결론" 흐름(완성도 지표 → MTBF·MTTR)으로 Bento Grid를 구성하면 사용자가 자연스럽게 신뢰도 맥락을 파악할 수 있다.
- 용어별 교정 빈도·원문 비표준 용어 비율·데이터 완성도 지표를 독립 카드로 흩어두지 말고 같은 블록 안에 배치해야 의사결정 흐름이 한 화면에서 완결된다.

## 2026-06-15 — FMS 포털 다음 단계로 어떤 기능이 현장에 가장 필요할 것 같아?
- `glossary` 테이블 스키마 확정이 번역 품질·MTBF 집계 정합성·폼 입력 노이즈를 동시에 해결하는 upstream 공통 인프라이므로, 설계 순서는 **① glossary 스키마 합의 → ② 폼 드롭다운 전환(입력 레이어 잠금) → ③ MTBF 집계 뷰** 순이어야 함 — 순서가 뒤집히면 집계 뷰가 오염된 `failure_code` 위에서 동작하는 구조적 결함이 생김.
- 드롭다운 fetch 구조를 붙이기 전에 팀 전체가 glossary 컬럼 설계를 합의하지 않으면, 나중에 스키마 변경 시 이중 작업이 발생하므로 UI 개발 착수 전 스키마 동결이 선행 조건임.

## 2026-06-15 — 각자 전문 분야에서 올해 가장 기억에 남는 작업이나 인사이트가 있어?
- DB 스키마 설계는 데이터 정규화 문제가 아니라 **팀이 어떻게 소통할 것인가를 구조로 굳히는 작업**이다 — `glossary` 테이블의 `field_key / label_ko / label_en / source_system` 4컬럼 구조 하나가 번역 기준·집계 카테고리·UI 드롭다운을 동시에 해결한 사례로 확인됨.
- 시스템 품질의 upstream은 입력 레이어(스키마 설계 결정)에 있으며, 번역·분석·UI 모두 "정의 단계를 어디서 잡느냐"의 문제로 수렴함.
- 설계 완성의 기준에 **검증 루틴(예: 배포 후 샘플링·대조 주기)이 명세에 포함**되어야 하며, 그 전까지는 upstream 설계가 실제로 작동하는지 보장할 수 없음.

## 2026-06-15 — 오늘 업무 돌아보면서 개선하고 싶은 점 하나씩 얘기해볼까?
- glossary 합의 → 스키마 설계 → UI 구현 순서를 팀 루틴으로 확립해야 함. 현재 각 레이어가 선행 합의 없이 독립 구현되어 번역·분석·개발 모두 동일한 입력 통제 문제에 반복 충돌하고 있음.
- `failure_code` 필드를 자유 텍스트에서 glossary 키값 기반 드롭다운으로 전환하는 파일럿을 우선 진행. 고통 지점이 집중되어 있고 번역·분석·UI 세 레이어 문제를 단일 변경으로 검증 가능.
