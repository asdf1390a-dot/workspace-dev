## 2026-06-08 — 예지보전(PdM) 도입한다면 어떤 데이터부터 모아야 할까?
- `started_at` / `resolved_at` 타임스탬프가 PdM의 핵심 선행 데이터이며, 이 두 필드 없이는 MTBF(평균 고장 간격) 계산 자체가 불가능하다.
- 현재 Supabase `bm_events` 테이블에 이미 데이터가 축적되어 있으므로, 용어 정제 + 타임스탬프 보완만 완료되면 새 센서 없이도 즉시 분석 착수 가능하다.

## 2026-06-10 — KPI 대시보드에 지금 없는데 있으면 좋겠다 싶은 지표가 있어?
- `bm_events` 테이블에 데이터가 충분히 쌓여 있으나 집계 뷰가 없어 **설비별 MTBF** 산출이 안 되고 있음 → 예방 정비 타이밍을 수치 기반으로 전환하려면 집계 뷰 생성이 선행 과제
- `failure_code`, `started_at` 등 핵심 필드의 **미입력률**을 KPI와 함께 노출해야 상위 지표(MTBF 등) 수치의 신뢰도를 동시에 판단 가능 — 지표 단독 표시는 데이터 품질 맥락을 숨김
- 평가자 의견 보강: 필드 완성도(채워짐 여부)와 **코드값 정확도**(올바르게 입력됐는지)는 별개 지표 — 완성도만 집계하면 MTBF·MTTR 계산의 실질 신뢰도를 과대 평가할 위험 있음

## 2026-06-10 — 인도 현장 엔지니어들이 공문서에서 자주 오해하는 표현이나 용어가 있어?
- `status` 필드에 "Pending" 같은 모호한 값이 구분 없이 입력되면 집계가 뭉개지므로, DB 값 레벨에서 세분화가 필요하다.
- "Breakdown" vs "Malfunction" 혼용은 BM 이벤트 데이터에서 실제로 확인된 문제로, 같은 고장이 두 카테고리로 쪼개지면 패턴 분석이 불가능해진다.
- 용어 글로사리 작성 시 "이 용어 = DB `failure_code` 필드의 어떤 값"까지 명시하지 않으면, 번역은 맞아도 데이터가 잘못 입력되는 케이스가 반복된다.

## 2026-06-10 — 우리 팀이 지금보다 더 잘 협업하려면 뭐가 바뀌어야 할까?
- 용어 통일 시 "이 용어 = DB의 어느 필드, 어느 값"까지 함께 정의해야 집계 오류를 방지할 수 있음
- 입력 폼이 자유 텍스트이면 동일 사건이 여러 카테고리로 분산되어 데이터 품질이 저하되므로, 드롭다운 고정이 용어 통일과 세트로 이루어져야 함

## 2026-06-13 — BM 이력 데이터 보면서 공장 설비 관리에 어떤 패턴이 보여?
- 용어 혼용 상태에서도 `started_at` 날짜 기반 시계열 분석은 노이즈 영향을 덜 받아 병렬 진행 가능하며, 모터류 설비는 고부하 시즌에 BM이 집중되는 패턴이 텍스트 정제 전에도 관찰됨
- `failure_code` 표준화 후 배포 한 달 시점에 실제 입력 기록 10건 샘플을 뽑아 선택된 코드값과 실제 고장 내용을 대조하는 검증 게이트가 필수 — 구조 변경이 데이터 품질 개선으로 이어졌는지 확인하지 않으면 드롭다운 교체에 그침

## 2026-06-13 — 모바일에서 FMS 포털 쓴다고 가정하면 어떤 화면이 제일 자주 열릴까?
- 모바일 접근 빈도가 높아질수록 `failure_code` 자유 텍스트 비율이 함께 상승하는 패턴이 있으며, 이는 데이터 품질 저하로 직결됨
- 드롭다운 강제화는 UX 개선이 아닌 데이터 품질 관리 문제로 분류해야 하며, 특히 모바일 환경에서 그 중요성이 높음
