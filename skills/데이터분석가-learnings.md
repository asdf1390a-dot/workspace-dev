## 2026-05-13 — 오늘 업무 돌아보면서 개선하고 싶은 점 하나씩 얘기해볼까?
- 보고서마다 컬럼명·단위 표기가 제각각이어서 집계 시 수작업 정리가 과다 발생함; Asset ID처럼 동일 장비가 문서마다 다른 식별자로 기록되면 데이터 매칭 자체가 불가능해짐
- 번역 글로사리를 데이터 필드 기준으로 맞춰두면 분석 정확도와 번역 품질을 동시에 높일 수 있음

## 2026-05-13 — 인도 현장 엔지니어들이 공문서에서 자주 오해하는 표현이나 용어가 있어?
- "Breakdown" vs "Malfunction" 용어 혼용 시 BM 이벤트 집계에서 동일 장비 사고가 다른 카테고리로 분류되는 데이터 오염 발생 — 글로사리에 "Breakdown = 가동 중단 수반" 같은 판단 기준을 데이터 필드 레벨로 명시해야 수작업 정제 부담을 줄일 수 있음
- 보고 양식 자체에 드롭다운으로 선택지를 고정하는 것이 용어 오해로 인한 데이터 분류 오류의 근본 해결책

## 2026-05-13 — 우리 팀이 지금보다 더 잘 협업하려면 뭐가 바뀌어야 할까?
- DB 집계 오류 방지: 같은 이벤트를 "Breakdown"/"Malfunction"처럼 혼용하면 동일 이벤트가 다른 카테고리로 분리되어 분석 결과가 틀어짐. 글로사리 합의 시 데이터 필드 기준을 함께 정의해야 분석 정확도를 보장할 수 있음.
- 글로사리 테이블을 단일 진실 공급원으로 활용하면 `bm_event_type` 등 이벤트 분류 필드의 카테고리 일관성이 유지되어 집계 쿼리 신뢰성이 높아짐.

## 2026-05-13 23:30 — [인사이트] manufacturing KPI dashboard best practices
검색 결과가 KPI 대시보드와 무관한 내용(뿌네 제조사 목록)으로 반환됐습니다. 제 자체 지식으로 작성합니다.

---

**제조 KPI 대시보드는 "보여주는 것"이 아니라 "결정을 빠르게 하는 것"이 목적이다** 📊

좋은 대시보드는 OEE·불량률·다운타임 3개 핵심 지표를 최상단에 두고, 드릴다운 구조로 원인 추적을 지원한다. FMS 포털에서도 BM 이벤트·설비 가동 데이터를 이 구조로 배치하면 현장 의사결정 속도가 높아진다. 지표가 많을수록 판단이 느려지므로 "지금 조치 필요한가?"에 즉시 답할 수 있는 알림 임계값 설정이 핵심이다.

→ 적용 포인트: FMS 대시보드에 OEE 기반 임계값 알림 추가 검토 🔧

## 2026-05-14 — 웹 UI에서 현장 작업자들이 가장 헷갈려할 것 같은 부분이 어디야? 개선
- 자유 텍스트 입력 필드는 동일 사건이 "Breakdown", "Malfunction", "Fault" 등 다양한 표현으로 입력되어 DB 집계 오류를 유발하므로, 드롭다운 고정 선택지로 대체하면 데이터 오염을 원천 차단할 수 있음
- 드롭다운 선택지 텍스트를 "BM (Breakdown Maintenance)" 형태로 풀어쓰면 작업자의 잘못된 분류 선택도 줄어들어 입력 단계에서 데이터 품질 확보 가능
- 개선 후 효과 측정을 위해 실제 작업자 제출 기록 10건 이상 랜덤 추출 spot check를 최소 한 달치 수행해야 구조 변경의 실질적 효과를 확인할 수 있음
