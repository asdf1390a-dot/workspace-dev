## 2026-05-13 — 인도 현장 엔지니어들이 공문서에서 자주 오해하는 표현이나 용어가 있어?
- "Breakdown" vs "Malfunction" 용어 혼용 시 BM 이벤트 집계에서 동일 장비 사고가 다른 카테고리로 분류되는 데이터 오염 발생 — 글로사리에 "Breakdown = 가동 중단 수반" 같은 판단 기준을 데이터 필드 레벨로 명시해야 수작업 정제 부담을 줄일 수 있음
- 보고 양식 자체에 드롭다운으로 선택지를 고정하는 것이 용어 오해로 인한 데이터 분류 오류의 근본 해결책

## 2026-05-13 — 우리 팀이 지금보다 더 잘 협업하려면 뭐가 바뀌어야 할까?
- DB 집계 오류 방지: 같은 이벤트를 "Breakdown"/"Malfunction"처럼 혼용하면 동일 이벤트가 다른 카테고리로 분리되어 분석 결과가 틀어짐. 글로사리 합의 시 데이터 필드 기준을 함께 정의해야 분석 정확도를 보장할 수 있음.
- 글로사리 테이블을 단일 진실 공급원으로 활용하면 `bm_event_type` 등 이벤트 분류 필드의 카테고리 일관성이 유지되어 집계 쿼리 신뢰성이 높아짐.

## 2026-05-14 — 웹 UI에서 현장 작업자들이 가장 헷갈려할 것 같은 부분이 어디야? 개선
- 자유 텍스트 입력 필드는 동일 사건이 "Breakdown", "Malfunction", "Fault" 등 다양한 표현으로 입력되어 DB 집계 오류를 유발하므로, 드롭다운 고정 선택지로 대체하면 데이터 오염을 원천 차단할 수 있음
- 드롭다운 선택지 텍스트를 "BM (Breakdown Maintenance)" 형태로 풀어쓰면 작업자의 잘못된 분류 선택도 줄어들어 입력 단계에서 데이터 품질 확보 가능
- 개선 후 효과 측정을 위해 실제 작업자 제출 기록 10건 이상 랜덤 추출 spot check를 최소 한 달치 수행해야 구조 변경의 실질적 효과를 확인할 수 있음

## 2026-05-15 — 각자 맡은 역할에서 '이건 AI가 못 대체하겠다' 싶은 부분이 있어?
- 숫자 이상치 발생 시 설비 결함인지 정기 점검 등 현장 이벤트인지 판단하려면 AI의 패턴 인식만으로는 부족하고 현장 맥락을 아는 사람의 해석이 필요하다.
- "Breakdown"과 "Malfunction" 같은 유사 용어 혼용으로 집계가 틀어질 때, 그것이 데이터 오류인지 의도적 구분인지 판단하는 최종 검증은 현장 맥락을 아는 분석가 몫이다.

3. 데이터베이스 검증 — FK 관계, 중복, 인덱스, 응답시간 (EXPLAIN 활용)
4. 비즈니스 로직 검증 — 계산 오류, 경계값, 예외처리
5. 최종 판정 — ✅ 승인 / 🟡 조건부 / 🔴 재검토

- Audit System (완료 예상 2026-05-20) — API + DB + cron 통합 검증

## 2026-05-16 — BM 이력 데이터 보면서 공장 설비 관리에 어떤 패턴이 보여?
- 같은 설비에서 특정 주기로 BM이 몰리는 집중 패턴이 존재하며, 모터류 설비는 우기 시즌·고부하 구간에 BM 빈도가 급증하는 경향이 있음
- 현재 FMS BM 데이터에서 가장 빠르게 꺼낼 수 있는 인사이트는 "어떤 설비가 반복 고장인가"보다 **"언제 고장이 집중되는가"** 이며, 시계열 분석 시 패턴이 즉시 가시화됨
- 데이터 신뢰성(용어 표준화)이 전제되어야 반복 고장 패턴 분석이 유효하며, 이는 집계·패턴 탐색보다 선행되어야 할 조건임

## 2026-05-16 — 재고/부품 관리에서 가장 자주 발생하는 문제가 뭘까? 자동화할 수 있을까
- BM 이벤트 완료 시점에 해당 부품 재고를 자동 차감하는 DB 트리거를 연결하면 수작업 입출고 기록 누락으로 인한 재고 수량 불일치를 대폭 줄일 수 있음
- 안전 재고 임계값(`stock_qty < min_stock`) 알림을 붙이면 부품 부족으로 수리가 지연되는 상황을 사전에 감지 가능
- `bm_events` 이력 집계 뷰를 `asset_id` 기준으로 만들면 설비별 자주 쓰는 부품 패턴을 데이터로 파악할 수 있으며, 이를 재고 예측 및 발주 자동화에 활용 가능
