**이 템플릿은 평가 작업 시작 시 자동으로 로드됩니다.**

## 🔴 Spot Check 최소 기준 (필수)

### 표본 크기
- **최소 표본 수**: 10건 이상
- **최소 기간**: 1개월 데이터
- **선택 방식**: 무작위 추출 (타임스탬프 기반)

### 검증 대상 3가지 필수 엣지케이스

1. **네트워크 지연 환경**
   - [ ] 버튼 클릭 후 2초 이상 지연 상황에서 중복 제출 방지되나?
   - [ ] 오프라인 상태에서 폼 저장 시 어떻게 되나?
   - [ ] 네트워크 복구 후 재시도 로직 작동하나?

2. **권한 미설정 상황**
   - [ ] 접근 권한 없는 사용자 → 403 에러 표시?
   - [ ] 로그인 만료 후 API 호출 → 401 에러 처리?
   - [ ] 읽기만 가능한 사용자 → 쓰기 버튼 비활성화?

3. **데이터 경계값 (Boundary Cases)**
   - [ ] 빈 데이터 상태 (empty array) → UI 표시 정상?
   - [ ] null/undefined 값 → 크래시 아님?
   - [ ] 매우 긴 텍스트 (1000자+) → UI 레이아웃 깨짐 아님?
   - [ ] 음수 값 가능한 필드에 실제로 음수 입력 가능?
   - [ ] 최댓값 초과 입력 시 제약 작동?

## 📋 검증 리포트 표준 양식

```markdown
# [프로젝트명] QA 검증 리포트 — YYYY-MM-DD

## 검증 개요
- 검증 기간: YYYY-MM-DD ~ YYYY-MM-DD
- 표본 크기: 10건 (무작위 추출)
- 데이터 출처: [테이블명] 타임스탬프 기반

## 5개 영역 검증 결과

### 1️⃣ 요구사항 일치도
- [ ] ✅ 완전 일치 (스펙 100% 구현)
- [ ] 🟡 부분 일치 (스펙 대비 미흡 항목 있음)
- [ ] 🔴 불일치 (주요 기능 누락)

**발견 사항:**
- 예: "BM 폼에 priority 필드 추가됨 (스펙 요구 ✅)"

---

### 2️⃣ API 응답 검증
- [ ] ✅ 정상 (타입, null 처리, 경계값 모두 정상)
- [ ] 🟡 부분 오류 (특정 필드에서 null 미처리)
- [ ] 🔴 실패 (타입 불일치 또는 크래시)

**검증 항목:**
- [ ] 응답 타입 일치? (expected vs actual)
- [ ] null/undefined 처리 정상? (UI 크래시 없음?)
- [ ] 경계값 처리 정상? (빈 배열, 최댓값 등)

**예시 테스트:**
```bash
curl -H "Authorization: Bearer $KEY" \
  "https://api/endpoint?limit=0&offset=1000" \
  # 경계값: limit=0 (무한 루프?), offset 초과 (에러 or empty?)
```

---

### 3️⃣ 데이터베이스 정합성
- [ ] ✅ 정상 (FK, 중복, 인덱스 모두 정상)
- [ ] 🟡 경고 (느린 쿼리 또는 부분 제약 누락)
- [ ] 🔴 실패 (FK 위반 또는 데이터 오염 발견)

**검증 항목:**
- [ ] Foreign Key 제약 작동? (관계 무결성)
- [ ] 중복키 제약 있나? (primary key, unique)
- [ ] 인덱스 설정? (EXPLAIN ANALYZE 확인)

**SQL 예시:**
```sql
-- 성능 확인
EXPLAIN ANALYZE SELECT * FROM table WHERE condition;

-- FK 확인
SELECT * FROM information_schema.table_constraints 
WHERE constraint_type = 'FOREIGN KEY';

-- 인덱스 확인
SELECT * FROM pg_indexes WHERE tablename = 'table_name';
```

---

### 4️⃣ 비즈니스 로직 검증
- [ ] ✅ 정상 (계산, 분기 로직 모두 정상)
- [ ] 🟡 부분 오류 (엣지케이스에서만 실패)
- [ ] 🔴 실패 (핵심 로직 오류)

**검증 항목:**
- [ ] 조건부 로직 정상? (if-else 모든 브랜치)
- [ ] 계산 정확? (요구사항 대비)
- [ ] 반복 로직 정상? (무한 루프 없음?)

**예시:**
```javascript
// 테스트 케이스
- BM 폼 제출 시 asset_id + failure_code 이미 존재?
  → 중복 방지 로직 작동하나?
- Priority = "Urgent" 선택 시 알림 발송되나?
- 1개월 이상 미해결 BM → 자동 에스컬레이션?
```

---

### 5️⃣ 최종 판정
- [ ] ✅ **승인** — 모든 항목 정상, 배포 가능
- [ ] 🟡 **조건부 승인** — 경미한 이슈만 남음 (배포 가능하나 보정 필요)
- [ ] 🔴 **재검토 필요** — 주요 결함, 배포 불가

## 발견 이슈

### 심각도별 분류

**🔴 Critical (배포 차단)**
- 데이터 무결성 위반 (FK 미충족, 중복 데이터)
- 기능 크래시 (null 참조 오류, 타입 불일치)
- 보안 결함 (권한 미인증, SQL injection 가능성)

**🟡 Medium (조건부 배포)**
- UI 레이아웃 이상 (매우 긴 텍스트에서만)
- 성능 저하 (1초 이상 지연, 하지만 폭주 아님)
- 문서 오류 (라벨 불일치, 약어 미정의)

**🟢 Low (배포 가능, 향후 개선)**
- UX 개선 기회 (사용성은 정상, 그러나 더 나은 방법 있음)
- 테스트 커버리지 부족 (핵심 경로는 정상)

## 데이터 품질 Spot Check (BM/재고 입력 양식용)

### BM 이벤트 기록 10건 샘플 검증

```
표본: 최근 1개월 BM 기록 10건 무작위 추출

[ ] 1. Asset ID 드롭다운 선택 정상?
      (수동 타이핑 불가? DCMI-*만 입력되나?)

[ ] 2. Failure Code 일관성 확인
      예: "모터 소손" + "Motor burnt out" 혼용 없나?
      → 모두 glossary 기준 표준 용어로 입력되나?

[ ] 3. Priority 필드 활용
      긴급도 높은 경우 "Urgent" 선택했나?
      → 우선순위 전달 희석 없나?

[ ] 4. 입력 완성도
      필수 필드(asset, failure_code, priority) 모두 채워졌나?
      → BM 폼 완성도 지표 계산 가능?

[ ] 5. 재현성 검증
      기록된 실패 내용이 현장 상황과 일치하나?
      (AI 추천을 무조건 선택한 건 아닌가?)
```
