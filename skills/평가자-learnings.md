## 2026-05-13 — 오늘 업무 돌아보면서 개선하고 싶은 점 하나씩 얘기해볼까?
- 글로사리-스키마-UI 정합성은 구축 이후 **검증 주기**가 없으면 drift가 발생한다. 월 1회 cross-layer spot check(번역 문서, DB 스키마, UI 라벨 3개 레이어 동시 비교) 루틴이 필요하다.

## 2026-05-13 — 인도 현장 엔지니어들이 공문서에서 자주 오해하는 표현이나 용어가 있어?
- 용어 오해 → 보고서 데이터 오염으로 이어지는 경로는 전형적인 결함 패턴: 드롭다운·글로사리 연동이 설계되어도 현장 엔지니어가 실제로 의도한 대로 선택하는지는 별도 검증 필요
- 용어 정의 변경 또는 신규 항목 추가 시 "구 보고서 기준"과 "신 기준" 간 해석 불일치가 발생하는 drift 패턴을 인지하고 관리해야 함
- 월 1회 샘플 보고서 spot check: 번역 문서·DB 데이터·UI 라벨 세 레이어를 동시 비교하는 루틴을 QA 체크포인트로 고정하는 것이 현실적인 방어선

## 2026-05-13 — 우리 팀이 지금보다 더 잘 협업하려면 뭐가 바뀌어야 할까?
- glossary SST(단일 진실 공급원) 구조에서 번역 문서·DB 데이터·UI 라벨 세 레이어 간 일치 여부를 월 1회 cross-layer spot check로 주기적으로 검증하는 루틴이 필요하다.
- 구조 설계 이후 "만들고 끝" 상태가 되면 레이어 간 drift가 발생하므로, 검증 주기 설계를 구조 설계 단계에서 함께 합의해야 한다.

FMS 포털처럼 실시간 데이터를 다루는 앱은 정상 흐름보다 엣지케이스에서 무너진다. 네트워크 끊김, 권한 없는 접근, 빈 데이터 상태 — 이 세 가지가 현장에서 가장 자주 터지는 구간이다. QA 체크리스트는 "되는가"가 아니라 "망가졌을 때 안전한가"를 기준으로 짜야 한다.

## 2026-05-14 — 웹 UI에서 현장 작업자들이 가장 헷갈려할 것 같은 부분이 어디야? 개선
- 드롭다운 선택지로 바꿔도 작업자가 선택지 의미를 이해하고 올바르게 고르는지는 별도 검증 필요 — 잘못된 선택지를 자신있게 고르는 것도 데이터 오염임
- 제출 후 확인 화면이 있어도 네트워크 지연 환경에서 버튼 중복 클릭으로 인한 중복 제출 여부를 별도로 테스트해야 함
- 구조 개선 후 효과 측정은 실제 작업자 제출 기록 10건 랜덤 추출 spot check를 최소 한 달치 수행해야 유효한 판단 가능

## 2026-05-15 — 각자 맡은 역할에서 '이건 AI가 못 대체하겠다' 싶은 부분이 있어?
- 결함 판단 기준 자체를 사람이 정해야 함: "버튼이 안 눌린다"는 피드백이 버그인지 의도된 제약인지 AI는 재현만 할 뿐 판단 불가 — 현장 맥락 없는 반복 검증은 무의미
- 결함의 귀속 범주 판단: 용어 혼용 문제가 보고서 데이터에 흘러들어올 때, 이것이 QA 결함인지 기획 결함인지 구분하는 것은 사람의 영역
