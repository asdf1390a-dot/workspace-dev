# 모델 선택 기준 표준화 (2026-05-20)

## 절대 규칙 (무조건 준수)

### 🟡 Haiku 4.5 (기본값)
**언제:** 대부분의 작업
- 채팅 응답 (모든 대화)
- 현황보고 (상태 리포팅)
- 지시 전달 (팀원 위임)
- 문서 읽기 (메모리/파일 조회)
- 간단한 코드 수정 (10줄 이하)
- 단순 검색/조회

**효율:** 비용 낮음, 속도 빠름, 정확도 충분

**금지:** Opus/Sonnet 대체 불가 — Haiku로 충분하면 Haiku만 사용

---

### 🔵 Sonnet 4.6 (선택적)
**언제:** 중간 복잡도 작업
- API 설계 리뷰 (구조 검토)
- DB 스키마 최적화 제안
- 코드 리팩토링 (30~100줄)
- 기술 문제 분석 (원인 파악)
- 신기능 설계 (간단한 규모)
- 문서 작성 (1000줄 미만)

**효율:** Opus 대비 비용 40%, 속도 3배, 복잡도 충분

**판단 기준:** "Haiku로 부족하지만 Opus 전체 능력 불필요"

---

### 🔴 Opus 4.7 (전문 작업 전용)
**언제:** 복잡도 높은 작업만
- 아키텍처 설계 (시스템 수준)
- 대규모 코드 리뷰 (100줄+)
- 복잡한 리팩토링
- 신기술 학습 + 교육
- 다중 도메인 문제 해결
- 동시 다중 작업 조정

**효율:** 비용 높음(Sonnet 대비 2배), 정확도 최고

**필수 조건:** 
- [ ] Haiku/Sonnet으로 불가능함을 확인
- [ ] 복잡도 근거 문서화
- [ ] 서브에이전트로 격리 실행 (메인 context 보호)

**금지:** 단순 작업에 Opus 사용 ❌

---

## 마이그레이션 기준 (언제 상향?)

| 현재 상태 | 판단 | 다음 모델 |
|---------|------|---------|
| Haiku 충분 | (유지) | Haiku 유지 ✅ |
| Haiku 부족 | Sonnet 시도 | → Sonnet |
| Sonnet 부족 | Opus 필수 | → Opus (서브에이전트) |
| 복잡도 불명 | 작은 작업부터 시작 | Haiku 시작 |

**규칙:** 아래쪽으로만 이동 (Haiku→Sonnet→Opus). 역방향 ❌

## 적용 사례

### ✅ Haiku 사용 (올바른 예)
- "현황보고" 요청 → Haiku (정보 리포팅)
- 메모리 파일 조회 → Haiku (읽기 작업)
- 팀원 지시 전달 → Haiku (텍스트 명령)
- 간단한 스크립트 작성 → Haiku (50줄 미만)

### ❌ Opus 오용 (틀린 예)
- "현황보고서 작성" → Opus 사용 ❌
- "파일 읽고 결과 보고" → Opus 사용 ❌
- "현재 상황 설명" → Opus 사용 ❌

### ✅ Sonnet 정당한 사용
- "DB 스키마 최적화 제안" → Sonnet (중간 복잡도)
- "API 설계 검토" → Sonnet (구조 분석)
- "코드 리뷰 50줄" → Sonnet (정확도 필요)

### ✅ Opus 정당한 사용
- "전체 아키텍처 재설계" → Opus (시스템 수준)
- "500줄 리팩토링 계획" → Opus (대규모)
- "3개 도메인 통합 설계" → Opus (복잡도 높음)
