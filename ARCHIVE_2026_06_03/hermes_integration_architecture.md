## 3. Integration Points

### 3.1 Data Ingestion (Hermes → Supabase)
**Read from:** `memory/active_work_tracking.md`, DSC FMS Supabase tables  
**Action:** Hermes cron jobs parse CTB, extract status snapshots  
**Output:** `/home/jeepney/.hermes/sessions/` + auto-generated reports

**Use Cases:**
- Daily 08:00: Fetch pending blockers → Send Telegram summary
- Daily 14:00: Parse Phase A checkpoints → Validate completion
- Daily 18:00: Generate team capacity utilization report

### 3.2 Task Delegation (OpenClaw → Hermes)
**Trigger:** When OpenClaw task hits "autonomous background loop"  
**Pattern:** Hermes skills handle pure information gathering (no writes to DB)  
**Output:** Results written to `/home/jeepney/.hermes/sessions/` → read by OpenClaw

**Example Flow:**
```
OpenClaw: "Need daily asset health report"
  ↓
Hermes: Run `asset-health-check` skill (scheduled)
  ↓
Output: JSON report → /home/jeepney/.hermes/sessions/asset-health-2026-05-19.json
  ↓
OpenClaw: Read file, format for Telegram, send to user
```

### 3.3 Messaging Coordination (OpenClaw ↔ Hermes)
**Telegram/Discord Integration:**
- OpenClaw remains primary message receiver
- Hermes: Read-only passive listener (no direct sends to user)
- Handoff: OpenClaw routes autonomous reports to user with Hermes origin tag

**Tag Format:**
```
【Hermes 자동리포트】
[Hermes Report]
└─ Generated by: [skill-name]
└─ Timestamp: 2026-05-19 08:15 KST
└─ Data: [report content]
```

## 8. Safety & Governance Rules

**Absolute Constraints:**
1. ❌ Hermes never writes to Supabase (read-only)
2. ❌ Hermes never sends direct messages (file-based output only)
3. ❌ Hermes never overwrites OpenClaw memory files
4. ✅ All Hermes outputs are auditable (timestamped logs)
5. ✅ User can disable any Hermes job at runtime

**Audit Trail:**
- Every Hermes job logs to `/home/jeepney/.hermes/logs/`
- Timestamp + input data + output file path recorded
- Weekly audit report generated by OpenClaw
