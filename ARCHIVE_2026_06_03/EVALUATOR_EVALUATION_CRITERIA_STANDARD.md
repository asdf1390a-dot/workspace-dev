## 📋 평가 4단계 오류분류 (Error Taxonomy)

### 🔴 Level 1: 설계오류 (Design-Level Errors) — 치명적, 구현 불가
**특징:** 시스템 아키텍처, 데이터 흐름, 사용자 워크플로우 레벨  
**감지 시점:** 설계 문서 리뷰 단계

**포함 범위:**
- DB 스키마 충돌 (동일 컬럼명 중복, 타입 불일치)
- API 라우트 충돌 (동일 경로 중복 정의)
- 보안 정책 위반 (RLS 미설정, JWT 미검증)
- UI/UX 불일치 (컴포넌트 구조 재설계 필요)
- 의존성 누락 (필수 라이브러리/마이그레이션 미포함)
- 성능 산술 오류 (10k 행 쿼리 N+1 로직으로 설계)

**PASS 기준:**
- [ ] DB 스키마: 기존 테이블과 충돌 없음 (information_schema 검증)
- [ ] API 라우트: 신규 경로가 기존 엔드포인트와 중복 없음
- [ ] 보안: RLS 정책 명시, 공개 엔드포인트만 인증 제외
- [ ] UI 구조: 기존 컴포넌트 체계와 일관성 있음
- [ ] 의존성: 모든 마이그레이션 파일 포함, 외부 라이브러리 명시
- [ ] 성능: 대량 쿼리(1000+) 환경에서도 O(n) 이하 보장

**FAIL 기준:** 위 항목 중 1개 이상 미충족 → **재설계 필수**

---

### 🟠 Level 2: 로직오류 (Logic-Level Errors) — 높음, 재구현 필요
**특징:** 함수/메서드 레벨 로직, 비즈니스 규칙 구현  
**감지 시점:** 코드 구현 검토 단계

**포함 범위:**
- 데이터 검증 누락 (NULL/undefined 체크 없음)
- 예외처리 부재 (naked try-catch, 에러 무시)
- 상태 관리 오류 (race condition, 트랜잭션 미보호)
- 비즈니스 로직 오류 (할인 계산 잘못, 권한 체크 틀림)
- 타입 오류 (문자열 ↔ 숫자 암묵적 변환)
- 경계값 처리 오류 (0, 음수, MAX_INT 미처리)

**PASS 기준:**
- [ ] 입력값 검증: 모든 사용자 입력에 NULL/타입 체크
- [ ] 예외처리: 모든 외부 호출(DB/API)에 try-catch + 로깅
- [ ] 트랜잭션: 다중 쓰기 작업 시 BEGIN...COMMIT...ROLLBACK 보호
- [ ] 비즈니스 로직: 명시된 비즈니스 규칙을 정확히 구현
- [ ] 타입 안전: TypeScript strict mode 통과
- [ ] 경계값: 0, 음수, MAX_INT 등 edge case 처리됨

**FAIL 기준:** 위 항목 중 2개 이상 미충족 → **재구현 필수**

---

### 🟡 Level 3: 보안/성능 (Security & Performance) — 중간, 최적화 필요
**특징:** 암호화, 인증/인가, 쿼리 최적화  
**감지 시점:** 통합테스트 단계

**포함 범위:**
- 암호화 미흡 (평문 저장, 약한 해시)
- 인증 우회 가능 (토큰 검증 부족)
- 권한 검사 누락 (관리자만 실행해야 하는 API 공개)
- 쿼리 성능 저하 (인덱스 미적용, JOIN 초과)
- 메모리 누수 (무한 루프, 이벤트 리스너 미해제)
- SQL 인젝션 가능 (동적 쿼리 미보호)

**PASS 기준:**
- [ ] 암호화: PII(이메일, 전화번호) bcrypt/SHA256 해시
- [ ] 인증: JWT 토큰 만료시간 설정, 리프레시 로직 구현
- [ ] 권한: 모든 POST/PUT/DELETE 엔드포인트에 RLS/role 체크
- [ ] 쿼리 성능: explain으로 seq scan 확인, 인덱스 생성됨
- [ ] 메모리: 장시간 실행 후 메모리 증가 <5%
- [ ] SQL 인젝션: 모든 동적 값을 parameterized query로 처리

**FAIL 기준:** 위 항목 중 3개 이상 미충족 → **보안감사 + 수정 필수**

---

### 🔵 Level 4: 통합/UX (Integration & UX) — 낮음, 미세조정 가능
**특징:** 사용자 경험, 에러 메시지, 엣지케이스 처리  
**감지 시점:** 사용자 시나리오 테스트

**포함 범위:**
- 에러 메시지 불친절 (기술 용어만, 해결방법 없음)
- 로딩 상태 피드백 없음 (사용자 대기 중임을 모름)
- 폼 입력 검증 피드백 부재 (invalid 상태만 표시)
- 엣지케이스 동작 불명확 (빈 리스트, 권한 없음 시 UI)
- 성능 체감 저하 (초기로딩 >3초, 검색 응답 >500ms)
- 접근성 미흡 (색상만 정보 표시, 키보드 네비게이션 불가)

**PASS 기준:**
- [ ] 에러 메시지: 기술용어 제거, 해결방법 2줄 이내 제시
- [ ] 로딩: 3초+ 작업에 진행바/스피너 표시
- [ ] 폼 입력: invalid 필드 highlight + hint text (예: "최소 8글자")
- [ ] 엣지케이스: 빈상태/오류상태/권한 없음 모두 UI 처리
- [ ] 성능: 초기로딩 <2초, 검색응답 <300ms (로컬 테스트)
- [ ] 접근성: WCAG AA 기준 (색상+라벨, Tab 키 네비게이션)

**FAIL 기준:** 위 항목 중 3개 이상 미충족 → **UX 개선 필요** (배포 후 개선 가능)

## ✅ 1회 평가 체크리스트 (Single-Round Evaluation Checklist)

평가자는 **Level 1 + Level 2 + Level 3**를 1회에 동시 검토합니다.  
**Level 4 (UX)**는 배포 후 모니터링에서 처리합니다.

### 평가자 역할분담 없음 (단일 평가자)
- 총 소요시간: 90분 (설계 60분 + 코드 30분)
- 체크리스트: 아래 30개 항목 순차 검증

### 🟢 PASS 판정 조건
- [ ] Level 1: 6/6 항목 통과
- [ ] Level 2: 6/6 항목 통과
- [ ] Level 3: 6/6 항목 통과
- **결과:** GO → 배포 진행

### 🔴 FAIL 판정 조건
- Level 1: 1개 이상 미충족 → **설계 재검토 필수** (1-2일)
- Level 2: 2개 이상 미충족 → **코드 재구현** (1-3일)
- Level 3: 3개 이상 미충족 → **보안감사 + 수정** (1일)
- **결과:** NO-GO → 피드백 + 재작업


### 시나리오 2: 표준화 1회 평가 (제안)
```
1회: 설계 + 로직 + 보안 동시 검토  
누적: 95%+ 신뢰도 (명확한 기준 + 체크리스트)
```
**장점:** 기준 명확 + 빠른 완료 + 재작업 최소화

### 시나리오 3: 표준화 2회 평가 (옵션)
```
1회 (평가자 A): Level 1 + 2 (설계 + 로직)  
2회 (평가자 B): Level 3 (보안 + 성능)  
누적: 98%+ 신뢰도 (전문화 분담 + 병렬처리)
```
