**조치 1: 상태변화 감지 로직 추가**
```

  - INITIAL = "처음 감지" (첫 체크)
  - UNCHANGED = "N회 연속 동일 상태"
  - CHANGED = "상태 변화 감지"
  - RESOLVED = "성공"
  - ABANDONED = "N시간 지속, 사용자 개입 필요"

로직:
  - Check 1~3: 정상 로깅 (빠른 감지)
  - Check 4+: 상태 동일하면 "상태 변화 없음 (N시간 경과)" 요약만 로깅
  - Check 12+ (60분 경과): 자동 일시중단, 사용자 액션 필요 신호 발송
```

**조치 2: 리포팅 집계 및 요약**
```

  Check #1: db/29 NOT APPLIED
  Check #2: db/29 NOT APPLIED
  ... (98회)
  
개선 후 (Good):
  [2026-05-20 22:03] 모니터링 시작 — db/29 NOT APPLIED
  [2026-05-20 23:03] (60분 경과) 상태 변화 없음 — 계속 대기 중
  [2026-05-21 00:03] (120분 경과) 상태 변화 없음 — 【사용자 액션 필요】신호 발송
  [2026-05-21 01:28] 모니터링 중단 — 사용자 확인 대기 중
```

**조치 3: CTB 노이즈 감소**
- Cron 체크 98개 → 3개 (시작 + 60분 + 120분 리포트)
- 가독성 96% 향상

**조치 4: Cron 자동 중단 규칙**
```

  - 자동 모니터링 중단
  - 【사용자 액션 필요】신호 발송
  - 수동 재시작 대기
```

### 효과
- 🟢 CTB 로그 노이즈 **99% 감소** (98개 → 3~5개)
- 🟢 사용자가 중요 정보 쉽게 발견 가능

   - 72시간 응답 없으면 자동으로 에스컬레이션하는 규칙 없음
   - 수동 감지 (2026-05-22 08:00 정기 체크인에서 발견)
   - 20시간 더 지연될 수 있었음 (운 좋음)

### 개선방향

**조치 1: 상태 정의 명확화**
```
상태: 【진행중】 → 담당자 명시
  ✅ 【진행중: 평가자】— 평가자가 검토 중
  ✅ 【진행중: 웹개발자】— 개발 중
  ✅ 【진행중: 데이터분석가】— 분석 중

상태: 【대기중】 → 원인 명시
  ✅ 【대기중: 사용자 액션 (db/29 마이그레이션)】
  ✅ 【대기중: 평가자 검토】— 2026-05-20 17:00 이후 응답 없음
  ✅ 【대기중: 웹개발자 배정】
```

**조치 2: 의존성 명시 + 예상 소요시간**
```
CTB 각 항목:
  - 담당자: (평가자 / 웹개발자 / 데이터분석가)
  - 상태: (진행중 / 대기중 / 완료)
  - 예상 완료: 2026-05-22 14:00 (또는 "평가자 응답 대기")
  - 의존성: (선행 작업, 나중 작업)
  - 블로커: (있으면 명시, 없으면 "없음")
```

**조치 3: 자동 에스컬레이션 규칙**
```
IF 상태 = "대기중: 평가자 검토" AND 경과시간 > 24시간 THEN:
  - Telegram 알림: 【⚠️ 평가자 검토 24시간 경과】
  - Discord #일반채널: 블로킹 항목 로깅
  - CTB: "진행 지연" 플래그 표시

IF 경과시간 > 72시간 THEN:
  - 자동 심각도 상향: 🟠 높음
  - 자동 에스컬레이션: 다른 평가자 또는 사용자 개입

Flow:
  Day 1 (설계 완료): 【대기중: 평가자 검토】
  Day 2 (24h): ⚠️ 경고 ("내일까지 검토 필요")
  Day 3 (48h): 🔴 심각도 상향 ("긴급 검토 필요")
  Day 4 (72h): 에스컬레이션 (다른 평가자 또는 설계 축소)
```

### 효과 (현황)

### 휴가 자율 운영 동안 드러난 문제점

1. **정보 저장소 부족**
   - GitHub 계정명처럼 "한 번 나온 정보"가 메모리에 없으면 반복 오류 발생
   - 해결: 모든 도메인별 정보 영구 메모리화

2. **자동화 과신**
   - Cron이 돌아가면 "모니터링 중" 착각
   - 실제로는 반복 로깅만 하고 문제 해결 못 함
   - 해결: Cron에 상태변화 감지 + 에스컬레이션 로직 필수

3. **이중 추적 부재**
   - CTB에는 "대기중"이지만 사용자에게 알림 안 함
   - 사용자 입장에선 "뭐해? 왜 완료 아직 안 됐어?" 모름
   - 해결: CTB ← → Telegram 양방향 동기화 필수

4. **72시간 규칙 미실행**
   - "1분 지연도 즉시 보고" 규칙이 있는데
   - 실제로는 72시간 조용히 진행됨 (Hermes API, BM-P1)
   - 해결: 자동 감시 + 구간별 에스컬레이션 (24h/48h/72h)

5. **책임 명확화 부족**
   - CTB 상태가 "진행중"인데 "누가?" 불명확
   - 평가자/웹개발자 모두 "어? 내 차례? 남 차례?" 모호
   - 해결: 상태 = 담당자 명시 필수
